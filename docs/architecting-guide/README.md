# アーキテクチャ設計の心得

## 大原則： 正解を探すのではなく、ベターな最適解を選び抜く！

- 正解のアーキテクチャは存在せず、その時点で最も良い（悪くない）ものを選ぶしかない
- 考慮すべき品質特性(-ility) を知り、トレードオフを踏まえて意思決定する
- すべての具体的なアーキテクチャは特殊解であり、それを考えるために一般解(アーキテクチャパターン) を知る
- 良い設計とは、状況の変化に合わせて変化（進化）させやすい設計である
- 「なぜこの設計にしたのか」という背景と根拠を記録し、将来の変更に備える

## [structure: どのようにシステムを構成するか](./01-structure.md)

コンセプト: システム全体を適切なコンポーネントに分割し、責任を明確にする

トピック:

- レイヤードアーキテクチャ: プレゼンテーション層、ビジネスロジック層、データアクセス層
- モジュール分割: 凝集度と結合度、境界づけられたコンテキスト
- データフロー: データの流れとライフサイクル

## [quality: どのような品質特性を実現するか](./02-quality.md)

コンセプト: トレードオフを理解し、システムに求められる品質特性(-ility)を実現する

トピック:

- スケーラビリティ: ステートレス設計、水平スケーリング、負荷分散
- 可用性: 冗長化、フェイルオーバー、Circuit Breaker
- パフォーマンス: キャッシング戦略、非同期処理、データベース最適化
- セキュリティ: 多層防御、ゼロトラストアーキテクチャ
- 保守性: 設定の外部化、Observability、ADR (Architecture Decision Records)

## [pattern: どのようなパターンで実現するか](./03-pattern.md)

コンセプト: 実績あるアーキテクチャパターンを活用し、車輪の再発明を避ける

トピック:

- アプリケーションパターン: MVC, MVVM, Clean Architecture, Hexagonal Architecture
- 分散システムパターン: マイクロサービス、イベント駆動、CQRS, Saga
- クラウドネイティブ: Twelve-Factor App、コンテナ化、サーバーレス
- データパターン: RDBMS vs NoSQL、イベントソーシング、データレイク
