# 要件定義の心得

## 大原則：本当に必要なものは何なのか考え抜く！

- 目的(Why) と手段(What/How) を分けて考え、まずは目的を深掘りして認識を揃える
- ある工程と前工程のトレーサビリティを確保する
- 悩んだら、前工程=目的に立ち戻って考える

## 前提： IPA 共通フレーム 2013 における用語の定義について

「IPA 共通フレーム 2013」において、下記のプロセスは明確に定義されている。

- `企画プロセス`: 経営戦略や事業計画に基づいて、システム化の目的、目標、範囲（スコープ）、およびその実現の可能性を決定するプロセス。
- `要件定義プロセス`: システムが満たすべき機能や性能を明確にし、すべてのステークホルダーの合意を得て文書化するプロセス。
- `システム開発プロセス`: 合意された要件に基づき、システムを設計、構築、テストし、受け入れ可能な状態にするプロセス。

一方で、 `要求` と `要件` については明確に定義されておらず、いずれも英語における `Requirement` の訳語として、同等の言葉として扱われている。

本ガイドにおいては、`要求` と `要件` を下記の通り使い分けることする。

- `要求`: 「企画プロセス」で扱われる `Requirement` に相当する用語として扱う
  - 主体: 事業側（発注者側）
  - 役割: 達成したい目的や課題、またはシステムに期待する機能やサービスとして提示するもの。抽象度が高いインプット。
- `要件`: 「要件定義プロセス」で扱われる `Requirement` に相当する用語として扱う
  - 主体: 技術側（受注者側）
  - 役割: 事業側の要求を分析・整理し、システムが満たすべき具体的、かつ検証可能な仕様として定義したもの。抽象度が低いアウトプット。

## [requirement: どのように要求を整理するか](./01-requirement.md)

コンセプト: **_漠然とした要望や問題意識を収集し、課題をシンプルに構造化してコンセプトを打ち出す_**

トピック:

- 洞察と目標設定: 今はまだ誰もわかっていない「本当に必要なもの」をあぶりだす
- コンセプト策定: 解決策をコンセプトとストーリーで語り、ビジネス価値を訴求する
- スコープ定義: やらないことを定めることで、やることを浮き彫りにする

## [business: どのように業務要件を定義するか](./02-business.md)

コンセプト: **_システムを取り巻く登場人物をモデル化し、システムの役割を浮き彫りにする_**

トピック:

- 用語（ユビキタス言語）の定義: 十人十色に使われる一般用語を、十人一色にする
- 登場人物（アクター）の定義: 人も機械もモデル化し、シンプルな世界として考える
- 業務フローとユースケースの定義: システムの外部の振る舞いを定めることで、システムに求められる振る舞いを浮き彫りにする
