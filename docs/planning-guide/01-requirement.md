# requirement: どのように要求を整理するか

**_漠然とした要望や問題意識を収集し、課題をシンプルに構造化してコンセプトを打ち出す。_**

---

## 関連キーワード

- **Why / What / How** (なぜ / 何を / どのように - Why から始める)
- **As-Is / To-Be** (現状とあるべき姿 - ギャップを課題として定義する)
- **KGI / KPI** (Key Goal Indicator / Key Performance Indicator - 目標達成度の測定指標)
- **MoSCoW** (Must / Should / Could / Won't - 優先順位付けの手法)
- **狩野モデル** (5 つの品質要素 - 優先順位付けの手法)
- **トレードオフスライダー** (優先順位をするかを可視化するためのツール)
- **QCD** (Quality:品質 / Cost:コスト / Delivery:納期 - トレードオフの 3 要素)
- **Elevator Pitch** (エレベーターピッチ - 短時間で簡潔に説明するプレゼンテーション手法)
- **Ethos Logos Patos** (エトス:倫理・道徳 / ロゴス:論理 / パトス:情熱・共感 - アリストテレスの弁論術)
- **MVP** (Minimum Viable Product - 実用最小限の製品)
- **Scope Creep** (スコープクリープ - 要件がなし崩し的に肥大化すること)
- **期待値コントロール** (相手に抱かれる期待を適切に管理・調整し、認識のずれや過度な期待による不満を防ぐ)

## 洞察と目標設定の心得

**_今はまだ誰もわかっていない「本当に必要なもの」をあぶりだす。_**

### 目的（Why）を深掘りする

- ユーザーの要望（Wants）の裏にある、本当の悩み（Pain）や本当に必要なもの（Needs）を探る
  - 例: 「データをエクスポートしたい」と言われたら、「なぜ？」と問う。「他システムへ連携したい」なら、エクスポートではなく API 連携が正解かもしれない
- 「要望」が手段（How）である場合、そのまま受け取らず、目的（Why）を深掘りし、Pain や Needs への理解を深める
  - 本当に必要なものは、誰もはっきりとイメージできていない（＝潜在ニーズである）可能性もあるため、深掘りすることで見出していく
- Pain や Needs から、目的を整理してビジネス要求（Why）として定義する

### 解決すべき課題を言語化し、目標（What）を定める

- ビジネス要求（Why）を起点に、現状（As-Is）とあるべき姿（To-Be）のギャップを言語化し言語化する
  - 現場の具体的事例や問題意識から現状を分析し、As-Is を整理する
  - 漠然とした To-Be について議論・言語化を繰り返し、関係者から共感を得られる形になるまで洗練させる
  - To-Be をイメージするのが難しい場合は、Why と As-Is がヒントとなる
- As-Is/To-Be のギャップをビジネス課題（What）と定義し、ビジネス課題を解決する（ギャップを埋める）ことで到達する状態（ゴール）を目標（What）として定義する
  - 目標は、測定可能な指標（KGI/KPI）に落とし込む
    - 定量目標（数値目標）は重要だが、定性目標（質的な目標、数値化できない目標）も同等以上に重要
    - 定性目標（質的な目標）は、評価基準を設けて測定可能にする工夫を行う

### 目標に優先順位をつける

- リソース(時間、労力、お金、等) は有限であることを前提に、課題・目標のビジネス価値（インパクト）と実現難度（リスク/コスト）を鑑みて優先順位を決める
  - 例: MoSCoW 分析 - 優先順位を 1 軸の 4 クラスに分類する
    - 対応必須（Must have）
    - 対応推奨（Should have）
    - 対応可能（Could have）
    - 対応先送り（Won't have）
  - 例: 狩野モデル - 品質を「充足-不充足」と「満足-不満」の 2 軸で、5 クラスに分類する
    - 当たり前品質要素 (Must-Be Quality Element)
    - 一元的品質要素 (One-Dimensional Quality Element)
    - 魅力的品質要素 (Attractive Quality Element)
    - 無関心品質要素 (Indifferent Quality Element)
    - 逆品質要素 (Reverse Quality Element)
- トレードオフスライダーを作成し可視化しておくと、今後の指針になる
  - 例: QCD + S（スコープ） のトレードオフスライダー
  - 例: 各種要求のトレードオフスライダー

## コンセプト策定の心得

**_解決策をコンセプトとストーリーで語り、ビジネス価値を訴求する。_**

### 一言で言えるコンセプトを作る

- 解決策が「誰の、どんな課題を、どのように解決するものなのか」をシンプルに言語化する（エレベーターピッチ）
- コンセプトには、解決策のユニークな価値（差別化ポイント）を含めることで、プロジェクトの独自性を明確にする。
  - 悪い例: 「多機能な営業支援ツール」
  - 良い例: 「移動中のスマホ操作だけで、日報入力が完了する営業支援アプリ」
- 迷いや衝突が生じた際、コンセプトという客観的な基準に立ち戻ることで、迷走を予防し、意思決定の速度を保つ
  - 例: 機能追加で迷ったとき、個人の意見でなく、「それはコンセプトに合致するか？」という観点で客観的に判断する

### 訴求ポイントを明確にする

- 提供するビジネス価値を明確に説明できるようにする
  - 価値を「機能提供」ではなく、「得られる便益」として表現する
    - 例: 「面倒な作業から解放され、本来の業務に集中できる」
  - 便益を得て、関係者が As-Is から To-Be に変わっていく様子を説明する
    - 例:
      - As-Is: 「帰社後の日報入力に毎日 1 時間費やしているせいで、残業が増え、家族との時間も削られている。」
      - To-Be: 「移動中に 5 分で入力が完了するため、定時退社を実現し、夜は子供とゆっくり過ごせるようになった。」
- 質的な変化を示すことで、プロジェクトの意義を訴求する
  - 2~3 倍程度の量的な変化は単なる効率化だが、 10 倍, 100 倍, 1000 倍のような圧倒的な変化は、質的な変化となる（量質転化）
- 論理的であるのみならず、感情にも訴えかけるようなストーリーで語れるとベター（エトス・ロゴス・パトス）

### 実現可能性を疑う

- コンセプトは魅力的であると同時に、実現可能性（フィージビリティ）を伴っている必要がある
  - 少なくとも、実現可能だという確信を与えられる必要がある
- 既存の技術的制約やビジネス制約を逸脱していないか、コンセプトを常に評価し、裏付けを持つ

## スコープ定義の心得

**_やらないことを定めることで、やることを浮き彫りにする。_**

### MVP を定義し最優先とする

- MVP として価値を提供できる最小限のコア機能セットを定義し、最も優先順位の高いスコープとする
- 装飾や便利機能は後回しにし、まずは「使える」状態を目指す
- 完璧を目指してリリースが遅れるより、早く出してフィードバックを得る
- 大規模に作り込む前に、プロトタイプや MVP で価値を検証する
  - 「作ってみたが使われない」リスクを最小化する

### 「やらないこと」を明記し期待値調整する

- 「やること」だけでなく、「やらないこと（Out of Scope）」を定義・明記し、関係者の期待値をコントロールする
  - 例: 「今回のリリースでは、決済機能は実装しない（請求書払いのみ）」
  - 明記しないと、ステークホルダーは勝手に「当然あるもの」と期待してしまうリスクがある

### トレーサビリティを確保し変更管理する

- 最初から完璧な目的・課題・目標・スコープを特定することできないという前提に立つ
- 状況の変化やプロジェクトの進展に応じて、目的・課題・目標・スコープも変化していくものだという前提に立つ
- 目的・課題・目標・スコープ間（Why/What/How 間）のトレーサビリティを常に確認・確保し、変更要求が発生した際には、変更管理プロセスを経ることで整合性を保つ
- 変更管理とともに、合意の証跡を確保することも心がける

### スコープクリープを防ぐ

- プロジェクト途中での追加要望（スコープ変更）は、納期やコストへの影響（トレードオフ）を踏まえて判断する
  - トレードオフスライダーや「やらないこと」を参考にする
