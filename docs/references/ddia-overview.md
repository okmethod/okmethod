## 第I部: データシステムの基礎

1台のマシン上でも分散環境でも共通する、データシステムの基礎知識。

### 1章:　信頼性・スケーラビリティ・メンテナンス性

データシステムを設計する上での3つの中心的な関心事を定義する。

- **信頼性（Reliability）** : 障害（ハードウェア故障・ソフトウェアバグ・人為的ミス）が発生しても、システムが期待どおりに動作し続ける能力。意図的に障害を起こしてシステムを鍛える「カオスエンジニアリング」の考え方とも接続する
- **スケーラビリティ（Scalability）** : 負荷の増大（データ量・リクエスト数・同時接続数など）に対して、性能を維持しながら対処できる能力。「負荷パラメータ」（例: Twitter のフォロワー数分布）と「パフォーマンス指標」（レイテンシのパーセンタイル・スループット）で定量的に議論する
- **メンテナンス性（Maintainability）** : 時間が経ってもシステムを扱う人々の生産性を維持すること。3つの観点に分解できる
  - **運用性** : オペレーターが日常業務（監視・デプロイ・障害対応など）をしやすい
  - **単純性** : 新規開発者がシステムを理解しやすい。偶発的複雑性を抽象化で取り除くことが重要
  - **進化性** : 要件変更に対してシステムを変えやすい。アジャイルやリファクタリングの考え方と接続する

### 2章:　データモデルとクエリ言語

アプリケーションがデータをどう表現・操作するかを規定するデータモデルの選択肢を比較する。

- **リレーショナルモデル** : テーブル・行・列・結合によるデータ表現。多対多の関係を正規化で表現しやすく、SQL による宣言的クエリが強力
- **ドキュメントモデル** : JSON/XML などの自己記述的な階層構造。スキーマ柔軟性が高く、データ局所性（関連データをひとまとめに読める）が利点。一方で結合処理が弱い
- **グラフモデル** : ノードとエッジで多対多の複雑な関係を自然に表現。ソーシャルグラフ・推薦システム・不正検知などに適する。クエリ言語として Cypher（Neo4j）や SPARQL がある
- **インピーダンスミスマッチ** : オブジェクト指向コードとリレーショナルモデルの間の変換コスト。ORM はこれを緩和するが完全には解消しない
- **宣言的クエリ vs 命令的クエリ** : SQL のような宣言的クエリはオプティマイザに最適化を委ねられるため、実装詳細を隠蔽しやすく、並列実行にも適する

### 3章:　ストレージと抽出

データベースがデータをどのように格納・検索するかの内部構造を掘り下げる。

- **ログ構造ストレージ（LSM-Tree / SSTable）** : データを追記専用のログに書き込み、バックグラウンドでコンパクションして SSTable（Sorted String Table）を生成する方式。書き込みスループットが高く、HBase・Cassandra・LevelDB などで採用される
- **Bツリー** : 従来型 RDBMS の主流。ページ単位でディスクを読み書きし、インプレース更新する。読み取りが速く、トランザクション的に扱いやすい
- **OLTP vs OLAP** : OLTP（Online Transaction Processing）は少量行を頻繁に読み書きするユースケース。OLAP（Online Analytical Processing）は大量行を集計するユースケース。アクセスパターンが異なるため、専用の最適化が必要
- **カラム指向ストレージ** : OLAP 向けに、列ごとにデータを格納する方式。集計クエリで不要列を読み飛ばせるため I/O を削減でき、圧縮効率も高い。Parquet・Redshift・BigQuery などで採用される
- **マテリアライズドビューとデータキューブ** : 集計結果を事前計算して保持することで、クエリの応答を高速化する手法

### 4章:　エンコーディングと進化

メモリを共有しないマシン間でデータを授受するための変換（シリアライズ/でシリアライズ）の方式と、スキーマ変更への対応戦略を整理する。

- **テキストフォーマット（JSON / XML / CSV）** : 人間可読で広く使われるが、型情報が曖昧でデータサイズが大きくなりやすい
- **バイナリフォーマット（Thrift / Protocol Buffers / Avro）** : スキーマ定義をもとにコンパクトかつ高速にエンコードする。フィールドタグや型情報を活用してスキーマ進化を安全に管理できる
- **前方互換性・後方互換性** : コードとデータのバージョンが混在する環境でローリングアップデートを可能にするための制約。古いコードが新しいデータを読める（前方互換）・新しいコードが古いデータを読める（後方互換）の両立が重要
- **データフローのモード** : データがシステム間を流れる経路は3種類ある
  - **データベース経由** : 書いたプロセスと読むプロセスが異なる時点でアクセスする。スキーマ互換性の管理が重要
  - **サービス経由（REST / RPC）** : 呼び出し側と提供側が同時に稼働する同期的なやりとり。API の互換性維持が必要
  - **非同期メッセージパッシング** : キューやブローカー（Kafka など）を介して間接的にデータを渡す。送信側と受信側が疎結合になる

## 第II部: 分散データ

複数ノードにデータを分散させる際に生じる固有の問題とその設計パターン。

### 5章:　レプリケーション

同じデータを複数ノードに複製することで、可用性・フォールトトレランス・読み取りスケーラビリティを高める手法。

- **シングルリーダーレプリケーション** : 1台のリーダーに書き込みを集約し、フォロワーに非同期/同期でレプリケートする方式。シンプルだが、リーダー障害時のフェイルオーバーに複雑さがある
- **マルチリーダーレプリケーション** : 複数のリーダーが書き込みを受け付ける方式。地理的に分散したデータセンター間の書き込み可用性を高められる一方、書き込み競合の解決が必要になる
- **リーダーレス（Dynamo スタイル）レプリケーション** : すべてのノードが書き込みを受け付ける方式。クォーラム（w + r > n）を用いて整合性を確保する
- **レプリケーションラグ** : 非同期レプリケーションでは、フォロワーがリーダーに追いつくまでの遅延が生じる。これを許容する設計を**結果整合性(=結果一貫性)**と呼ぶ。ラグが引き起こす異常を防ぐための一貫性保証として以下がある
  - **自分の書き込みを読む（Read-Your-Writes）** : 自分が書き込んだデータは必ず自分が読めることを保証する。書き込み直後に別レプリカを読んで変更が消えたように見える異常を防ぐ
  - **単調読み取り** : 一度読んだデータが次の読み取りで「巻き戻って」見えないことを保証する。同じユーザーが常に同じレプリカを読むなどの方法で実現する

### 6章:　パーティショニング

データセットを複数のパーティション（シャード）に分割し、複数ノードに分散させることで書き込み・読み取りのスループットを向上させる手法。

- **レンジパーティショニング** : キーの範囲でパーティションを分割する方式。レンジスキャンが効率的だが、ホットスポットが生じやすい
- **ハッシュパーティショニング** : キーのハッシュ値でパーティションを決定する方式。データを均等に分散できるが、レンジクエリには弱い
- **セカンダリインデックスのパーティション** : ドキュメント単位（ローカルインデックス）かタームごと（グローバルインデックス）で管理する方法があり、書き込みと読み取りのトレードオフが異なる
- **リバランシングと動的パーティション** : ノード追加・削除に際してパーティションを再分配する方法。固定数パーティションの事前割り当て vs 動的分割の2アプローチがある
- **リクエストルーティング** : クライアントがどのノードにリクエストを送ればよいかを知る仕組み。ZooKeeper などのコーディネーションサービスやゴシッププロトコルが使われる

### 7章:　トランザクション

複数の読み書き操作をひとまとめにして原子的に扱う仕組みと、それに伴うトレードオフ。

- **ACID** : トランザクションが満たすべき4つの性質。特に「分離性」は実装コストが高く、さまざまなレベルが存在する
  - **原子性（Atomicity）** : トランザクション内の操作はすべて成功するか、すべて失敗してロールバックされるかのどちらか
  - **一貫性（Consistency）** : トランザクション前後でデータの整合性制約が満たされた状態を保つ。アプリケーション側の責務でもある
  - **分離性（Isolation）** : 並行して実行されるトランザクションが互いに干渉しない。強度は分離レベルで調整する
  - **永続性（Durability）** : コミットされたデータは障害後も失われない。WAL（Write-Ahead Log）などで保証する
- **Read phenomena（読み取り異常）** : 分離レベルが不十分なときに発生する読み取り系の問題。SQL 標準で定義された3種類と、それ以外の実務上重要な問題がある
  - **ダーティリード** : 別トランザクションが未コミットの書き込みを読んでしまう現象。Read Committed 以上で防止
  - **ノンリピータブルリード** : 同一トランザクション内で同じ行を2回読むと、他トランザクションのコミットにより結果が変わってしまう現象。Repeatable Read 以上で防止
  - **ファントムリード** : 同一トランザクション内で同じ条件の範囲クエリを2回実行すると、他トランザクションの挿入・削除により結果件数が変わってしまう現象。Serializable で防止
  - **ロストアップデート** : 2つのトランザクションが read-modify-write を並行して行うと、一方の更新が消えてしまう現象。Repeatable Read + SELECT FOR UPDATE や atomic 操作で防止
  - **ライトスキュー** : 2つのトランザクションがそれぞれ異なる行を更新するため競合検出されないが、組み合わせると整合性制約を破る現象。ロストアップデートの一般化形。Serializable でのみ防止できる
- **分離レベル** : 分離性の強度を段階的に定義したもの。強いほど安全だが、並行性が低下しパフォーマンスコストが高くなる
  - **Read Committed** : ダーティリード・ダーティライトを防止する最低限の分離レベル。多くの RDBMS のデフォルト
  - **スナップショット分離 / Repeatable Read** : トランザクション開始時点のスナップショットを読む。ノンリピータブルリードを防止。MVCC で実装されることが多い
  - **シリアライザブル** : すべての競合状態（ファントムリード・ライトスキューを含む）を防止する最強の分離レベル。実用上は 2PL・直列実行・SSI のいずれかで実装する
- **シリアライザブル分離の実装** : シリアライザブルを実現するアプローチは3種類ある
  - **2相ロック（2PL）** : 読み書きにロックを取得し、トランザクション終了まで保持する悲観的手法。デッドロックが発生しうる
  - **直列実行** : トランザクションを単一スレッドで順番に処理することで競合をそもそも排除する。スループットに上限があるが、シンプルで確実
  - **シリアライザブルスナップショット分離（SSI）** : スナップショット分離をベースに、コミット時に競合を検出してアボートする楽観的手法。並行性を維持しながらシリアライザブルを実現できる

### 8章:　分散システムの問題

分散環境特有の障害モードと、それがシステム設計に与える根本的な制約。

- **部分障害** : 分散システムでは一部のコンポーネントだけが故障することがあり、故障しているかどうかの判定すら困難。単一マシンとは根本的に異なる障害モデル
  - **信頼性のないネットワーク** : パケット遅延・ロスト・重複が起こりうる。タイムアウトで障害を検知するが、適切な閾値設定は難しい
  - **信頼性のないクロック** : 分散ノード間で時計は完全には同期されない。「壁時計」（実時刻）と「単調クロック」（経過時間計測）の使い分けが重要。NTP によるズレ補正も誤差を消せない
  - **プロセスの停止** : GC ポーズ・仮想マシンのサスペンド・OS スケジューリングにより、プロセスが任意のタイミングで一時停止しうる。リース（時間制限付きロック）とフェンシングトークンで古い権限の行使を防ぐ
- **障害の前提モデル** : ノードがどのような壊れ方をするかを前提として明示したもの。前提を変えるとアルゴリズムの設計難易度が大きく変わる
  - **クラッシュストップフォールト** : ノードは故障したら完全に停止し、以後一切の応答をしないと仮定する。最も単純なモデル
  - **クラッシュリカバリフォールト** : ノードはクラッシュしても再起動して処理を再開しうると仮定する。再起動後も永続化されたデータは保持される。現実のサーバ障害に近いモデルで、Paxos や Raft はこれを前提とする
  - **ビザンチン障害** : ノードがクラッシュするだけでなく、誤った・悪意ある応答を返す可能性まで仮定する。最も過酷なモデル。ブロックチェーンなど高い信頼性が必要な環境で考慮される
- **タイミングの同期モデル** : 分散アルゴリズムを設計・評価する際の前提となる、ネットワーク遅延やクロックのずれに関する仮定
  - **同期モデル** : メッセージ遅延・クロックずれ・処理時間にそれぞれ既知の上限があると仮定する。現実のネットワークには当てはまらないが、理論証明の基礎として使われる
  - **部分同期モデル** : 多くの場合は同期的に動作するが、ときどき上限を超えることがあると仮定する。現実の分散システムに最も近いモデル
  - **非同期モデル** : 遅延・ずれに一切の上限を仮定しない最も厳しい前提。このモデルで正しく動くアルゴリズムは環境に依存しない堅牢な設計と言える

### 9章:　一貫性と合意

分散システムにおける一貫性保証の強弱と、分散合意の理論的・実装的基盤。

- **一貫性モデル** : 分散システムがデータの読み書きに対してどこまで整合性を保証するかを定義したもの。強いほど安全だが、可用性・レイテンシとのトレードオフが大きくなる
  - **即時一貫性（線形化可能性 / Linearizability）** : レプリカが単一コピーかのように振る舞う最強の保証。任意のタイミングで読んでも常に最新値が返る。CAS などのアトミック操作を安全に実装できる一方、高レイテンシ・低可用性とのトレードオフがある
  - **因果一貫性（Causal Consistency）** : 因果関係のある操作を正しい順序で観測できることを保証する。因果関係のない操作の順序は保証しない。線形化可能性より弱いが、可用性を保ちながら直感的な整合性を提供できる。ランポートクロック・バージョンベクターが因果順序の追跡に使われる
  - **結果一貫性（Eventual Consistency）** : 更新が止まれば最終的にすべてのレプリカが同じ値に収束することを保証する。収束までの間は古い値が返ることを許容する。可用性とパフォーマンスを最優先する設計で採用される
- **CAP 定理** : 「一貫性・可用性・分断耐性のすべてを同時に満たすシステムは存在しない」という定理。ただし実際の設計では「CP か AP か」という二項対立よりも、レイテンシと一貫性のトレードオフとして捉えるほうが実践的
- **分散トランザクションと合意** : 複数ノードにまたがるトランザクションをコミットするために **2相コミット（2PC）** が用いられるが、コーディネーター障害時にブロックする問題がある。**Paxos / Raft / Zab** などのコンセンサスアルゴリズムがより堅牢な合意を実現する

## 第III部: 導出データ

既存データから新たな価値（ビュー・集計・予測など）を導出するシステムの設計パターン。

### 10章:　バッチ処理

有界なデータセットを一括処理するバッチジョブの設計原則と実装パターン。

- **Unix 哲学とデータパイプライン** : 「単一機能のツールを組み合わせてパイプでつなぐ」という Unix の思想は、大規模バッチ処理でも有効な設計指針となる
- **MapReduce** : 入力データを分割し Map（キーバリュー変換）・シャッフル（キー別集約）・Reduce（集計）の3フェーズで処理する分散バッチフレームワーク。Hadoop MapReduce が代表例
- **結合処理のパターン** : 分散バッチにおける結合の主要な実装パターンは3種類ある
  - **マージ結合** : 両側のデータを同じキーでソートしておき、順番にスキャンして結合する。大規模データ同士に向く
  - **ブロードキャスト結合** : 小さいテーブルをすべてのノードに配布し、大きいテーブルの各パーティションとローカルで結合する
  - **パーティション結合** : 両側を同じキーでパーティショニングし、対応するパーティション同士をノード内で結合する
- **データフローエンジン** : Apache Spark・Flink・Tez などは MapReduce の中間結果をディスクに書かずメモリ上で受け渡すため高速化できる。DAG で処理グラフを定義する

### 11章:　ストリーム処理

無限に流れてくるイベントをリアルタイムに継続処理するシステムの設計。

- **イベントストリームとメッセージングシステム** : Kafka のようなログベースのメッセージブローカーは、メッセージを耐久性をもって保持し、コンシューマが任意のオフセットから再生できる。これにより、バッチ処理とストリーム処理を統一的に扱える
- **変更データキャプチャ（CDC）** : データベースのレプリケーションログを読み取ってイベントストリームに変換する手法。サービス間のデータ同期や検索インデックス更新などに活用される
- **ストリームとデータベースの関係** : ストリームをデータベースのクエリ結果（マテリアライズドビュー）を継続更新するプロセスとみなすと、両者は本質的に同じものを異なる時間スケールで扱っていると理解できる
- **時間とウィンドウ** : **イベント時刻**（データが生成された時刻）と**処理時刻**（システムが受け取った時刻）のズレへの対処が重要。**タンブリングウィンドウ**・**スライディングウィンドウ**・**セッションウィンドウ** などの集計ウィンドウがある

### 12章:　データシステムの将来

本書の知見を総括し、信頼性・公平性を備えたデータシステムの設計指針を示す。

- **データ統合とデータフロー** : 組織内の複数システムを連携させるアプローチとして、「ETL でデータウェアハウスに集約」よりも「イベントログを中心としたデータフロー（Lambda / Kappa アーキテクチャ）」が変更に強い構造となる
- **導出データの冪等性と再処理** : バッチ・ストリームのどちらであっても、再処理可能な設計（冪等な変換・入力の保持）を採用することで、障害回復・スキーマ変更・バグ修正に柔軟に対応できる
- **正確性と非同期性** : エンドツーエンドの正確性（一意制約・デュプリケート排除）は、ストレージレイヤーのトランザクションだけでは保証しきれない。アプリケーションレベルの冪等性キーやリクエスト ID が重要
- **書き込みパスと読み込みパス** : データの加工・集計をどのタイミングで行うかの設計判断。**書き込みパス**（データが届いた時点で事前計算する）にコストをかけると読み込みが速くなり、**読み込みパス**（クエリが来た時点で計算する）に委ねると書き込みが軽くなる。インデックスやマテリアライズドビューは「書き込みパスにコストを前払いして読み込みを高速化する」設計の典型
- **倫理・公平性** : データシステムは人の意思決定に影響を与えるため、モデルのバイアス・プライバシー・説明責任の観点を設計段階から組み込む必要がある

## 所感・考察: DDIA から着想したアナロジー

### 単一マシンである個人と、分散システムである組織

- 人間は、単一の物理マシンと驚くほど似ている。
  - CPU:脳、メモリ:短期記憶、ストレージ:長期記憶、電源:体力、再起動:一眠りする
- 人間の組織は、複数マシンからなる分散システムと驚くほど似ている。
  - ノード:メンバー、ネットワーク:コミュニケーション経路、リーダー:意思決定者、フェイルオーバー:引き継ぎ・代理
- 個人の頭の中では全コンテキストが自然に共有されているが、組織ではメンバー間でメモリは共有されない。
  - 他人の脳には直接アクセスできないため、シリアライズ（言語化）とメッセージパッシング（コミュニケーション）が設計上の核心になる
  - このコーディネーションコストこそが「なぜ組織は個人より意思決定が遅くなるのか」の構造的な答えでもある
- 個人はスケールアップ（自己研鑽）に上限があるが、組織はスケールアウト（採用・分業）で理論上は無限に拡張できる。ただし、ノードが増えるほどオーバーヘッド（コーディネーションコスト）も増大する。

### 情報共有のトレードオフ

- 線形化可能性（全ノードが常に最新の情報を持つ）を実現しようとすると、コミュニケーション（会議等）が増えてしまい、正確かもしれないが遅い。
- 逆に、各自が手元の情報で動く結果整合性スタイルは速いが、認識ずれが発生し手戻り（ロールバック）のリスクがあがる。
- 組織が大規模になるほどネットワーク分断（部門間のサイロ化）は避けられない。CAP定理的に言えば、「全員が最新情報を持つ（一貫性）」と「いつでも素早く動ける（可用性）」を同時に満たすのは構造的に難しい。

### バッチ処理的なメール文化、ストリーム処理的なチャット文化

- メール文化はバッチ処理的と言える。
  - 長所: 1通あたりの情報量が多く、整理されている。必要な往復回数は少ない。
  - 短所: レスポンスが遅い。いきなり長文を送りつけられるとギョッとする。
- チャット文化はストリーム処理的と言える。
  - 長所: レスポンスが早い。テンポよく都度やりとりした方が方向性のずれを早期検知できる。
  - 短所: 1メッセージあたりの情報量が小さく、情報が不完全なことが多い。何往復も必要になりがち。
- メール・チャットという形態というよりも、送信(書き込みパス)にコストをかける設計か、受信(読み込みパス)にコストをかける設計か、と捉えた方が本質的かもしれない。
  - 教訓として、意図せず相手にコストを押し付けないようにする配慮を忘れてはならない。
- どちらが良いかは案件の性質や人同士の相性次第、という点も同じ。

### 悲観ロックと楽観ロック

- 悲観ロック的なアプローチは「事前に根回しと承認を取ってから動く」スタイルと言える。競合（利害衝突）が多い環境では安全だが、ロック待ちのため構造的に遅くなる。
- 楽観ロック的なアプローチは「まず動いて、コンフリクトが起きたら修正する」スタイルと言える。手戻りの発生確率が構造的に高くなるため、競合が少ない・失敗コストが低い環境では効率的だが、ロールバックのリスクを受容する必要がある。また、副作用のある処理（外部への情報発信等）はロールバックできない。
- どちらが適切かは環境次第だが、ライトスキュー的な問題（各自が「誰かがやるだろう」と判断した結果、誰もやらない）が起きやすい組織は楽観ロック過多のサインかもしれない。

### SPOF（単一障害点）と属人化

- 「その人がいないと回らない」という状態は、SPOF そのもの。シングルリーダーレプリケーションのリーダーが突然落ちた状態 = キーマンの退職・急病であり、フェイルオーバー設計（後任への切り替え）や、日頃からのレプリケーション設計（情報共有・ドキュメント化）の有無が障害復旧スピードを決める。
- 個人の頭の中にしかない知識は、書き込まれていない揮発性メモリと同じ。永続化（ドキュメント化）して初めてシステムとして信頼できる状態になる。
