## 「利用」と「頒布」

OSS に関わる義務は、「利用」か「頒布」かによって大きく異なる。

- **「利用」とは** : 組織内のサーバー上での実行、開発環境での利用、自社サービスとしての利用（SaaS 提供など）
- **「頒布」とは** : ソフトウェアのダウンロード販売、実行ファイルの提供、デバイスへの組み込み・販売

ライセンスによって、「頒布」にあたる条件に差異があるため注意。  
（例: AGPL は、ネットワーク経由でのサービス提供にも頒布に準じたコピーレフト義務が発生する、等）

---

## 「利用」時の基本的な心構え

- **保証の免責の理解** : OSS は無保証であり、欠陥があっても作者は責任を負わないことを理解し、利用者自身がリスクを引き受ける。

---

## 「頒布」時の義務

### 基本的な義務（ほぼすべてのライセンス共通）

- **帰属表示義務** : 著作権表示とライセンス条文を、製品やドキュメントに含める。
- **改変の明示義務** : OSS のソースコードを改変した場合、オリジナルのコードと区別できるように変更点を明示する。

### ライセンス伝播の義務

ライセンスの種類に応じて、派生成果物を同ライセンスで提供する。

- **コピーレフト**
  - コピーレフトライセンスの OSS を含むソフトウェアを頒布する場合、そのソースコードを同ライセンスの OSS として公開することを義務付ける。
  - (代表例) `GPL`, `AGPL`, etc
- **準コピーレフト**
  - "改変"されていない準コピーレフトライセンスの OSS を含むソフトウェアを頒布する場合、その OSS のオリジナルソースコードを入手できることを保証することを義務付ける。
  - "改変"された準コピーレフトライセンスの OSS を含むソフトウェアを頒布する場合、改変した部分のソースコードを公開することを義務付ける。
  - 成果物全体の公開義務は原則ないが、リンク方法によるため注意が必要。
  - (代表例) `MPL`, `LGPL`, etc
- **非コピーレフト**
  - 上記のようなソースコードの公開義務はない。
  - (代表例) `MIT`, `BSD`, `Apache License 2.0`, etc

## TIPS

### 言語ごとのライセンスリスト作成ツールチェーンまとめ

- [pip-licenses](https://github.com/raimon49/pip-licenses)
  - ユースケース: Python依存パッケージのリストを作りたい場合
  - リリース物だけのリストを作りたい場合: 本番依存のみをインストールした仮想環境を作成してから実行する（例: `uv sync --no-dev` 等）

- [license-checker](https://github.com/davglass/license-checker)
  - ユースケース: npm依存パッケージのリストを作りたい場合
  - リリース物だけのリストを作りたい場合: `npx license-checker --production` オプションを使用する

- [licensed](https://github.com/licensee/licensed)
  - ユースケース: 複数言語をまとめて同じ仕組みで管理したい場合
  - Github Actions の Composite Action として提供されているため、ワークフローに組み込む
  - かつては `github/licensed` だったが、 `licensee/licensed` になった模様
