# ソフトウェアエンジニアのためのセキュリティ知識体系まとめ

セキュリティエンジニア（専門家）ではない、一般のソフトウェアエンジニアが押さえておくべきセキュリティの知識をまとめる。

セキュリティ対策は、攻撃手法を個別に知り対策するだけではなく、セキュリティガバナンスの知識体系を踏まえ、指針とする必要がある。  
何を守るか・どのリスクを許容するか・インシデントが起きたときにどう動くか、というのは組織レベルの問題であり、エンジニアの実装はその指針の具体化である。

- **管理・ガバナンス編**: ISMS（情報セキュリティマネジメントシステム）の JIS Q 27001 (ISO/IEC 27001) の知識体系をベースに、判断基準や指針となる概念をまとめる。本資料で扱う。
- **実装・エンジニアリング編**: 「管理策」を具体的な「コード・設計」に落とし込む知識をまとめる。詳細は [web-security-overview.md](./web-security-overview.md) 参照。

## 情報セキュリティの 3 要素

情報のセキュアな管理方法を設計する上では、以下の3要素が評価軸となる。

- **機密性 (Confidentiality)**: 許可された者だけが情報にアクセスできること。
- **完全性 (Integrity)**: 情報が正確で、改ざん・破壊されていない状態にあること。
- **可用性 (Availability)**: 許可された者が、必要なときに情報および情報システムを利用できる状態にあること。

## サイバーセキュリティフレームワークの 6 コア

実装フェーズでは「防御」の観点が重要となるが、運用設計としては他フェーズの観点も求められる。

- **統治（Govern）**: 組織のサイバーセキュリティリスクマネジメント戦略、期待、ポリシーを確立し、伝達、監視（モニタリング）する。
- **識別（Identify）**: 組織の現在の守るべき資産（ソースコード、個人情報、サーバー、等）とサイバーセキュリティリスクを識別し把握する。
- **防御（Protect）**: 組織のサイバーセキュリティリスクを管理するためのセーフガード（セキュリティ対策）を使用する。
- **検知（Detect）**: 起こり得るサイバーセキュリティ攻撃及び侵害（不正アクセス）の可能性を発見し、分析する。
- **対応（Respond）**: 検知されたサイバーセキュリティインシデントに関するアクションを実行する。
- **復旧（Recover）**: サイバーセキュリティインシデントの影響を受けた資産及び業務が復旧する。

**参考**:

- [NIST サイバーセキュリティフレームワーク（CSF）2.0](https://www.ipa.go.jp/security/reports/oversea/nist/ug65p90000019cp4-att/begoj9000000d400.pdf)

## リスクマネジメント

何をどこまで守るか、どの程度のコストをかけるか、等の優先度付けを判断するためにはリスクマネジメントの観点が重要となる。  
エンジニアにも、リスクアセスメントに基づくリスク対応策・設計判断が求められる。

### リスクアセスメント

- **リスク特定**: どの資産に、どのような脅威（攻撃・ミス）と脆弱性があるかを洗い出す。
- **リスク分析**: 発生確率と影響度（機密性・完全性・可用性の喪失度）を算定する。
- **リスク評価**: 分析結果を基準と比較し、対策の優先順位を決める。

### リスク対応

- **リスク低減**: 技術的対策（暗号化、WAF 導入等）によりリスクを下げる。
- **リスク受容**: 対策コストがリスクを上回る場合、あえて現状維持とする。
- **リスク回避**: リスクのある活動自体を止める（例：危険な機能の廃止）。
- **リスク共有**: 保険加入や業務委託により、他者とリスクを分担する。

## インシデント管理策

ISO/IEC 27001 では、インシデント管理策を「予防・検知・是正」の 3 種別で整理する。

- **予防**: 情報セキュリティインシデントの発生を予防することを意図した管理策
- **検知**: 情報セキュリティインシデントの発生時に機能する管理策
- **是正**: 情報セキュリティインシデントの発生後に機能する管理策

### インシデント対応フロー

| フェーズ        | 内容                                   |
| --------------- | -------------------------------------- |
| 1. 検知・報告   | 異常の発見・記録・関係者への初動報告   |
| 2. 初動対応     | 被害拡大の防止（封じ込め）・証跡の保全 |
| 3. 根本原因分析 | 侵入経路・影響範囲の特定               |
| 4. 復旧         | システム・サービスの正常状態への回復   |
| 5. 事後対応     | 再発防止策の策定・教訓の文書化         |

### CSIRT（Computer Security Incident Response Team）

インシデント対応を専門に担う組織内チーム。平時から体制・手順・連絡経路を整備しておくことが前提となる。

- **社内 CSIRT**: 自組織のインシデントを検知・対応・調整する
- **外部連携**: [JPCERT/CC](https://www.jpcert.or.jp/) などの公的機関や業界 ISAC と情報共有する

### BCP / DR との関係

インシデントが事業継続に影響する規模になった場合、**BCP（Business Continuity Plan：事業継続計画）** に基づく対応に移行する。IT システムに限定した復旧計画は **DR（Disaster Recovery）計画** と呼ぶ。

- **RTO（Recovery Time Objective）**: 許容できる最大停止時間
- **RPO（Recovery Point Objective）**: 許容できる最大データ損失範囲（どの時点まで戻るか）

## 脆弱性とは

脆弱性（セキュリティホール）とは、プログラムや設定上の問題に起因するセキュリティ上の「弱点」。  
より具体的には、「想定した用途以外の動作の実行可能性」であり、悪用されるリスクがある。

### 基本指標

- **[CVE（Common Vulnerabilities and Exposures）](https://www.cve.org/)**: 個別脆弱性に付与される一意の識別番号
- **[CWE（Common Weakness Enumeration）](https://cwe.mitre.org/)**: 脆弱性の種類（タイプ）の分類体系
- **[CVSS（Common Vulnerability Scoring System）](https://www.ipa.go.jp/security/vuln/CVSS.html)**: 脆弱性の深刻度を 0.0〜10.0 で数値化する評価システム

### CVSS の 4 指標グループ

ベンダーが公表する基本値は攻撃環境や自組織の状況を考慮しない汎用スコアであるため、優先度判断には脅威値・環境値を組み合わせて評価する必要がある。

- **基本値 (Base)**: 脆弱性自体の技術的特性。ベンダーが公表し、時間経過によって変化しない。
- **脅威値 (Threat)**: PoC・攻撃ツール・攻撃事例の有無など、評価時点での脅威状況。v3.x の「現状値 (Temporal)」に相当。
- **環境値 (Environmental)**: 自組織のネットワーク構成・保有情報の重要度を加味したスコア。同じ脆弱性でもシステム環境により大きく変動する。
- **補足値 (Supplemental)**: 安全性・回復可能性・自動化可能性・価値密度など補足的な文脈情報。スコアには影響しない。

### 対応フロー

1. **Step1 絞込み**: 自組織に関係するソフトウェアの脆弱性情報のみを収集
2. **Step2 危険度確認**: CVSS基本値・PoC公開有無・攻撃事例の有無を確認
3. **Step3 影響分析**: 現状値・環境値を自組織で算出し、対策優先度を判断
