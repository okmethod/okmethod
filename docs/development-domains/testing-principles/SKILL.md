# テスト設計の心得

## 目的

テスト設計・テスト戦略・テスト計画の意思決定をサポートする。  
「プロダクションとテストは表裏一体」を前提に、**何をどの層で検証するかのトレードオフを踏まえた設計**を支援する。

## アセット

このスキルのナレッジファイルは `assets/` に配置されている:

- **`assets/01-approach.md`** — テスト戦略・計画・層定義・自動化方針
- **`assets/02-layer.md`** — 層割り当て・テストダブル・委譲設計
- **`assets/03-verification.md`** — カバレッジ・自動テスト維持・手動テスト運用・テストデータ管理
- **`assets/layer-assignment-cheatsheet.md`** — 層割り当てチートシート
- **`assets/documentations-cheatsheet.md`** — テストドキュメンテーションチートシート
- **`assets/coevolution-case-study.md`** — アーキテクチャとテストの共進化 ケーススタディ

---

## 発動タイミング（TRIGGER）

以下のキーワード・文脈が含まれる場合に発動する:

- テスト設計・テスト戦略・テスト計画の相談・レビュー
- テストピラミッド / テストトロフィー / テスト層構成の検討
- 単体テスト・結合テスト・E2E テストの範囲・役割分担の相談
- モック・スタブ・フェイク等のテストダブルの使い分け判断
- CI/CD へのテスト組み込み・自動化方針の検討
- テストコードの保守性・フレーキーテスト対策
- テスト仕様書・カバレッジマップの設計

## スキップ条件（SKIP）

以下の場合はこのスキルを使用しない:

- アーキテクチャ設計・技術スタック選定（→ アーキテクチャ設計の心得を参照）
- コーディング実装・リファクタリングの具体的な方法（→ コーディングの心得を参照）
- 要件定義・ユーザーストーリー作成（→ 要件定義の心得を参照）
- 特定テストフレームワークの API の使い方

## 自律実行ポリシー

- **コンテキストから推論できる場合は自律判断する** — ユーザーへの確認は最小限にとどめる
- **確認が必要な場合** — 回答の方向性が変わる制約条件が不明な場合のみ質問する
- **質問はまとめて 1 ターンで行う** — 必要な確認事項は一度にまとめ、往復を最小化する（最大 4 問）

## 実行手順

### Step 1: 問いを分析する

コンテキストから自律推論する。推論できない場合のみユーザーに確認する（1 ターンにつき最大 4 問）:

1. **対象スコープ**: システム全体 / 単一サービス / 単一コンポーネント（コード・文脈から推論）
2. **問いの階層**（キーワードから自律推論。判断が難しい場合は独自用語（Approach/Layer/Verification）を使わず確認する。例：「テスト全体の方針についてですか？特定の層の設計についてですか？」）:
   - **Approach** — テスト戦略・計画・層定義・自動化方針。変更コスト: 大。キーワード: テストピラミッド、テスト戦略、自動化方針、CI/CD、シフトレフト
   - **Layer** — 各層の境界・割り当て・テストダブル設計。変更コスト: 中〜大。キーワード: 単体テスト/結合テスト/E2E の境界、モック、スタブ、層割り当て
   - **Verification** — 検証フロー・カバレッジ・テストデータ管理。変更コスト: 小〜中。キーワード: カバレッジ、フレーキー、テストデータ、手動テスト、テスト仕様書
3. **制約条件**: 既存スタック・チーム規模・プロダクト成熟度（不明な場合のデフォルト前提: 中規模 Web サービス、小〜中チーム）

### Step 2: 対応アセットを読み込み、関連セクションを特定する

問いの階層に応じたファイルを **Read ツールで** 読み込む。複数の階層にまたがる場合は、**変更コストの大きい上位階層から順に**（Approach → Layer → Verification）読み込む:

| 問いの階層 | 読み込むファイル |
|---|---|
| Approach（戦略・計画・自動化方針） | `assets/01-approach.md` |
| Layer（層割り当て・テストダブル） | `assets/02-layer.md` |
| Verification（カバレッジ・運用・テストデータ） | `assets/03-verification.md` |

チートシートは以下の条件で参照する:

- 層割り当て・どの層で検証するかの判断が求められた場合 → `assets/layer-assignment-cheatsheet.md`
- テスト仕様書・ドキュメント構成の相談が出た場合 → `assets/documentations-cheatsheet.md`
- アーキテクチャとテストの共進化の実践例が求められた場合 → `assets/coevolution-case-study.md`

**読み込んだ後の処理**:

1. ユーザーの問いと照合し、関連するセクション・トピックを特定する
2. 関係ないセクションは読み飛ばし、関連箇所のみ抽出する
3. 抽出した内容を Step 3〜4 の回答根拠として使う

### Step 3: 関連トピックを提示する

**問いが 1 つのトピックに絞られている場合は Step 3 をスキップして直接 Step 4 へ進む。**

複数トピックにまたがる場合のみ、アセット内から関連するトピックを **2〜4 件** ピックアップして提示する。  
選定基準: **① 直接関連するもの優先、② 変更コストの大きい階層（Approach > Layer > Verification）を先に、③ 最大 4 件に絞る**。  
ユーザーが深掘りしたいトピックを選べるようにすることが目的。

**トピックマップ（変更コストの大きい順に並べている）**:

| キーワード例 | 参照トピック | ファイル |
|---|---|---|
| テスト戦略 / テスト計画 / 品質目標 | テスト戦略とテスト計画の心得 | `01-approach.md` |
| テストピラミッド / テストトロフィー / 層定義 | テスト層の定義と設計の心得 | `01-approach.md` |
| 自動化 / CI/CD / 実行環境 | 自動化方針の決定の心得 | `01-approach.md` |
| 仕様書 / テストコード / ドキュメント | 仕様書とコードの棲み分けの心得 | `01-approach.md` |
| 単体テスト / 結合テスト / E2E / 層境界 | 層割り当ての判断基準 | `02-layer.md` |
| モック / スタブ / フェイク / テストダブル | テストダブル・実物の使い分け | `02-layer.md` |
| テスト委譲 / 検証スコープ外 / 設計記録 | 委譲の記録の心得 | `02-layer.md` |
| カバレッジ / テストマトリクス / 網羅性 | カバレッジマップの設計 | `03-verification.md` |
| フレーキー / 不安定 / 自動テストの維持 | 自動テストの維持 | `03-verification.md` |
| 手動テスト / チェックリスト / テスト実施 | 手動テストの運用 | `03-verification.md` |
| テストデータ / フィクスチャ / 再現性 | テストデータの管理 | `03-verification.md` |

### Step 4: トレードオフを列挙して推奨を出す

- 選択肢を 2〜3 個挙げる
- 各選択肢の「得るもの」と「失うもの」を明示する
- 「最善解」ではなく「**最悪回避**」の観点で推奨を出す

### Step 5: 意思決定を記録する

以下のいずれかに該当する場合にテスト方針サマリを生成する:

- ユーザーが明示的に記録作成を依頼した場合
- Step 4 で Approach 階層のテスト戦略など、後戻りコストの大きい決定について、ユーザーが受諾・確定を示す発言をした場合

「なぜこの層構成にしたか」と「何を各層で検証しないと決めたか」をセットで残す。

## 設計判断の軸

1. **共進化させる** — プロダクション設計とテスト設計は同時に育てる
2. **変更コストの大きい決定から先に** — Approach > Layer > Verification の順で整理する
3. **最悪回避** — 最善解より「後悔しにくい選択」を優先する
4. **「何をテストしないか」も決める** — 委譲と除外の判断を明記する
