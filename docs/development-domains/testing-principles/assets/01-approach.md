# approach: どのような戦略でテストの全体像を計画するか

**コンセプト**: テスト全体像を定め、プロダクト構造に合わせてテスト層と自動化戦略を設計する

---

## 関連キーワード

- **Test Strategy** (テスト戦略 - テスト全体の方針の文書化)
- **Shift Left** (シフトレフト - テストをより早い工程に移動する考え方)
- **Test Pyramid** (テストピラミッド - 下層ほど多く書くテスト構造の指針)
- **Test Trophy** (テストトロフィー - インテグレーションテストを重視するモデル)
- **Unit Test** (単体テスト - 外部依存をまたがない、振る舞い単位のテスト)
- **Integration Test** (結合テスト - 外部依存をまたぐ複数コンポーネントの連携テスト)
- **E2E Test** (エンドツーエンドテスト - ユーザー操作を起点に、UI操作を通してシステム全体を検証するテスト)
- **Scenario Test** (シナリオテスト - UI操作に限らず、運用保守も含めた業務オペレーション全体を検証するテスト)
- **Regression Test** （リグレッションテスト・回帰テスト - デグレード検知のためのテスト）
- **Smoke Test** (スモークテスト - デプロイ後の基本動作確認)
- **Flaky Test** (フレーキーテスト - 不安定に成功・失敗を繰り返すテスト)

---

## テスト戦略とテスト計画の心得

**_「何を作るか」と「どう検証するか」をセットで考える。_**

### テスト戦略をシステム要件の一部として定義する

- テスト戦略は「品質をどう担保するか」の基本方針であり、開発開始前に決める
  - 要件定義で定めた業務シナリオ（導入、主要機能、運用保守、等）が、そのままテストシナリオの骨格になる
  - 要件の優先度（Must/Should/Could）が、そのままテストの優先度となる
  - この段階では層の詳細やテストコードの設計は不要 - 後工程での判断基準となる指針を定めることが目的
- テスト戦略を後付けにせず、プロダクション設計と並行で進める（Shift Left）
  - テスト戦略が決まると、プロダクション設計にも制約が生まれる
    - 例: 「ユニットテスト可能な構造にするため、外部依存は DI で注入する」
    - 例: 「外部 API 連携を持つため、コントラクトテストをインテグレーション層に追加する」
    - 例: 「リリース頻度が高いため、E2E はスモークテストのみに絞り、ユニット・インテグレーションで品質を担保する」
  - テスト戦略を後工程に回すと、要件と検証の間にギャップが生まれたり、そもそも検証できない要件が生まれるリスクがある
- すべてをテストすることは不可能なので、リスクの高い領域を優先する（リスクベーステスト）
  - リスクは「発生確率 × 影響度」で評価し、高いほどテスト層を厚くし、低いほど割り切る
  - 例: 決済・認証など障害時の影響が大きい機能はテスト層を厚くする
  - 例: 管理画面の参照系は正常系の手動確認のみで割り切る

### テスト計画は非エンジニアとも合意し、テスト設計はエンジニアの領分とする

- テスト計画（スコープ、スケジュール、体制、受け入れ基準、等）はビジネスサイドと合意する文書とする
  - 技術詳細を知らなくても判断できる「何を・いつ・誰が」を合意の対象にする
  - テストでのトラブル発生時や、リリース判定の際に説明しやすい粒度にしておき、情報共有の土台となる
- 一方、それを実現するためのテスト設計（層構成、モック方針、ツール、CI構成、等）は技術的決定であり、エンジニアの裁量で判断する
  - この分離により、「何を検証するか」を定めた上で、「どうやって検証するか」を柔軟に進化させられる
  - ビジネスサイドに技術詳細の承認を求めると、判断が遅くなる上に責任の所在が曖昧になる
- 両者の境界線は「リリース判断に関わるか」で引く
  - リリース判断に関わる（受け入れ基準・スコープ・スケジュール）→ ビジネスと合意
  - 技術的な実装の選択（ツール・技法・自動化方法）→ エンジニアが決定

### テストは何ショット繰り返すかを検討のうえ計画する

- テストはリリース前に問題を検知することが目的であり、1回通して終わりということはなく、必ず問題が見つかる
- 特に、1回目のテストは手順・スクリプト・コードの不備が大量に見つかる恐れがある
  - それはテストが失敗したというよりも、成果物が未完成だったということに近い
  - 1回目を「テスト期」と位置付けると、テスト期なのにまだ開発しているという認識のずれが生じるリスクがある
  - 1回目をいかに高速に回り、問題を早期に検知するかが、後続の検証の安定性を高める
  - テストオペレーションの自動化は、「初めの1回」を高速化できることの効果が特に大きい
- テストの規模に応じて、何ショット繰り返すか？をあらかじめ計画する
  - 1回目は問題の洗い出し、2回目はバグ解決確認＋追加改修分の確認、3回目は問題ないこと確認、などの位置付けも考えておくと良い
  - 最後の「問題ないこと確認」で問題が見つかることもあるので、もう1回テストを回せるバッファを置いておけるとスケジュールの安全性が高まる

### テストとプロジェクトライフサイクル

- 適切なテストは、プロダクトのフェーズによっても変わる
  - 開発初期: 高速なフィードバックを最優先にする
    - 単体テスト + スモークE2Eで頭とお尻を抑え、E2E自動化は仕様が固まってから進める
    - 仕様変更への追随を優先し、テスト構造を固めすぎない
  - 機能拡張期: 結合テストを拡充してE2Eの肥大化を防ぐ
    - カバレッジマップを整備し、機能追加のたびに観点の漏れをレビューできる体制を作る
    - 単体・結合テストのCI化を進め、フィードバックループを短縮する
  - 安定運用期: 変更の安全性とデグレード検知を確保する
    - 主要シナリオのE2E自動化と性能ベースラインを確立する
    - CI実行時間を管理し、テストスイートの棚卸しを定期的に行う
  - リプレイス期: テストスイートを仕様書・移行基準として活用する
    - 既存テストを新実装の検証に流用でき、テスト仕様書が移行仕様書を兼ねる
    - テスト構造を環境・フレームワークに縛られない状態に整えておくと移植しやすい
- テスト戦略・テスト設計は固定するのではなく、プロダクトの変化に合わせて進化させる
  - 見直しのシグナルの例:
    - モックの修正が頻発する: Over-Mocking の転換点（結合テストへの移行を検討）
    - CI が遅くなり開発者のローカル実行が疎かになる: テスト分布・実行戦略の見直しサイン
    - E2E 自動テストが頻繁に失敗し原因特定に時間がかかる: 対象を最重要シナリオに絞り詳細を下層に委譲
    - テストが通るのに本番でバグが発生する: テスト環境と実環境の乖離（環境パリティを見直す）

## テスト層の定義の心得

**_プロダクトのコンポーネント構成に基づいて、テストピラミッドの層を設計する。_**

### プロダクト構成を先に理解してからテスト層を決める

- プロダクトのコンポーネント構成（フロントエンド、バックエンド、データベース、外部サービス、等）に基づき、特性と境界とに合わせてテスト層を調整する
  - 例: BE単体 → BE-DB結合 → FE-BE結合 → BE-外部サービス結合 → E2E → 業務シナリオ
- 層の数が多いと管理コストが上がり、少なすぎるとギャップが生まれる - プロダクトの複雑度に比例させる
  - 「単体テスト → 結合テスト → E2Eテスト」の3層は出発点に過ぎない
- 「テスト層の名前」より「テスト層の責務」を先に定義する
  - 「単体テスト」「結合テスト」の目的・範囲は人によってイメージがずれやすいため注意する

### テスト分布は「形」でなく、「考え方」を参考にする

- テストピラミッド（単体テストを最多に）とテストトロフィー（結合テストを最重視）は、テスト分布の「形」について異なる主張をしているが、参考にすべきはそれぞれの背景にある考え方である
  - ピラミッドの考え方: 高速で安定したテストを土台にしないと、テストのコストをコントロールしきれない
  - トロフィーの考え方: モックに頼り過ぎると、テストが通っても実際の動作を保証できない
- テスト分布の形はプロダクトのアーキテクチャとテスト層の設計から「結果的に決まる」ものであり、目標として先に決めるものではない
  - ただし、テスト層の分布の偏りは、プロダクションコードの設計に問題がある可能性を疑う
    - 例: ユニットテストが書けない → 関数に複数の責務が混在している
    - 例: E2E でしか確認できない → レイヤー分離ができていない

### 層ごとの責務を明示する

- 各テスト層が「何を確認する層か」と「何を確認しない（委譲する）層か」を明示する
  - 例: 「BE-DB結合の層は CRUD 処理を検証する。ビジネスロジックの網羅は BE単体の層に委ねる」
  - これが曖昧だと、テストの重複や漏れの判断ができない
- 各テスト層が「どこで実行されるか」「何が前提・必要となっているか」を明示する
  - 例: 層1-3 はローカル、層4-5 は DEV環境（GPUが必要）、層6は STG環境、等
  - 実行環境の制約が、テストの自動化可否や実施頻度を規定する
  - 環境パリティ原則: テスト環境はできる限り本番と同じエンジン・スキーマを使う - 代替 DB や代替構成を使うと、本番固有の挙動（型の厳格さ・制約等）との乖離リスクがある
  - ローカルで実行できない層はタグ・マーカーで明示し、CI のみで動かす設計にする
    - 例: `@pytest.mark.integration` のようなマーカーを付与し、ローカルでは除外、CI では全実行
  - Docker Compose 等でローカルの実行環境を CI に近づけ、「ローカルでも動く層」を増やす

## 自動化方針の心得

**_各層の自動化レベルと実行環境を決め、CI/CD に組み込む。_**

### 自動化の判断基準を持つ

- テストピラミッド下層は原則自動化し、CIで毎回実行する
- テストピラミッド上層は、自動化する前に「手動でやる意味があるか」を問う
  - 手動に意味がある例: 業務シナリオテスト（手順書通りにオペレーションできるか？自体が検証対象）、UI/UXテスト（触った時の違和感など、テスト項目に起こせない点に気づけるため）
  - 自動化すべき例: リグレッションテスト、スモークテスト（毎回人手でやるにはコストが高いため）
- テストの自動化は「書くコスト」だけでなく、「維持するコスト」がかかる
  - 画面設計が頻繁に変わるフェーズで E2Eテストを自動化すると、メンテナンスの方が高くつく
  - フレーキーテスト（不安定テスト）を放置すると、テストへの信頼を毀損する
- 自動化コストと保守コストを天秤にかけ、手動テストの方が ROI が高い領域を見極める
  - 例: 一時的な仕様のテスト、UI 表示の細かなレイアウト確認は手動の方が安い場合がある
  - 例: 繰り返し実行される回帰テスト、境界値検証は自動化の優先度が高い
  - ただし、手動確認の証跡が義務付けられている場合（医療系・金融系・公共系の規制要件等）、自動化しても手動確認と証跡記録を並行して行う必要がある - 自動化の ROI 計算に「手動確認の代替」ではなく「並走コスト」を前提に含める
- 自動化を先走らず、まず手動テストで検証観点を固めてから自動化を検討する

### CI/CD の実行タイミングと環境を設計する

- テスト層ごとに実行タイミングを使い分ける
  - 例: ユニットテスト → PR 時に全実行（ローカルでも高速に動く）
  - 例: インテグレーションテスト → main マージ時または夜間バッチで実行（DB などが必要）
  - 例: 自動 E2E テスト → ステージング環境へのデプロイ後に実行
- テストが失敗した場合のマージブロック条件を明確に定める
  - 例: ユニットテスト失敗は PR マージ禁止。E2E は警告にとどめる（フレイキーリスクを考慮）

## 仕様書とコードの棲み分けの心得

**_テスト工程の成果物の解像度を上げ、役割分担・棲み分けを設計する。_**

### テスト仕様書は「観点の網羅性」、テストコードは「ケース詳細」を担う

- テスト仕様書に書くものの例
  - 検証観点の一覧（何を検証しているか）
  - カバレッジマップ（どの層で何をカバーしているか）
  - 手動テストの手順・期待結果（チェックリスト）
  - テスト間の設計判断（層の委譲理由等）
- テスト仕様書に書かないものの例
  - ユニットテストの全ケース一覧 - テストコードが正
  - テストクラス名・関数名の対応表 - コードと二重管理になる
  - テスト実施結果・日付 - テスト仕様書はクラスで、実施記録はインスタンス
- テストコードは具体的なテストケース（入力値・期待結果）の「正」である
  - 境界値・エッジケース（null、空配列、最大値など）はコードで表現するのが適切
  - 具体的・機械的・網羅的にテストケースを洗い出すプロセス自体が、仕様書の曖昧さ、コードとの齟齬を浮かび上がらせる

### テスト仕様書はレビューのために存在する

- テスト仕様書の最大の価値は、観点の抜け漏れをレビューできること
  - テストコードだけでは、テストが書かれていない観点の漏れに気づけない
  - そもそもなぜそのテストをするのか？というトレーサビリティを確保する必要があるが、テストコードから要件を追うのは難しい
- テスト仕様書のカバレッジマップを定期的にレビューし、機能追加時等に更新する
  - 要件 → テスト仕様 → テストコード、と中継点を設けてそれぞれマッチングすることで、要件とテストのトレーサビリティを確保する
  - テスト仕様書を見れば観点の抜け漏れを確認でき、コードを見れば具体的なケースを確認できる、という分業にする
