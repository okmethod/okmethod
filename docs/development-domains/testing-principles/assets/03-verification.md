# verification: どのように検証フローを運用するか

**コンセプト**: テスト仕様書を「レビュー可能な観点一覧」として維持し、テスト実施の運用フローを確立する

---

## 関連キーワード

- **Test Coverage** (テストカバレッジ - コードや機能の網羅率)
- **Exploratory Testing** (探索的テスト - 仕様書に基づかない直感的な探索テスト)
- **Test Fixture** (テストフィクスチャ - テストに必要な初期データや状態)
- **Test Data Factory** (テストデータファクトリ - テストデータを動的生成するパターン)
- **Seed Data** (シードデータ - テスト実行前に投入する定義済みのデータ)
- **Test Evidence** (エビデンス・テスト証跡 - テスト実施の記録)
- **Traceability** (トレーサビリティ - 要件からテストへの追跡可能性)

---

## カバレッジマップの心得

**_テスト層 × 検証観点のマトリクスで、網羅性を俯瞰する。_**

### カバレッジマップを「検証観点の地図」として整備する

- テスト層 × 検証観点のマトリクスを作成し、「どの観点が、どの層でカバーされているか」を一目で確認できる状態にする（参考: [カバレッジマップ テンプレート](./documentations-cheatsheet.md)）
- カバレッジマップには「何を確認するか」の観点レベルを書く - 具体的な入力値・期待結果はテスト仕様書に書く
- カバレッジマップはコードカバレッジ（行/分岐カバレッジ）とは別物と理解する
  - コードカバレッジ 100% でも「仕様の観点」が抜けていれば品質は保証されない
  - 例: 正常系のコード行は全部通っても「残高不足時の処理」観点が漏れていることはある
- 新機能追加時は、カバレッジマップを更新してからテストコードを書く

### 観点を先に分類してからケースを洗い出す

- 観点のカテゴリ（機能仕様 / 非機能 / 運用 等）を先に決めてからケースを洗い出すと、重複・漏れに気づきやすい
  - 参考: [テスト設計書の構成例](./documentations-cheatsheet.md)
- カテゴリを固めずに進めると、似た観点を複数の層で重複検証したり、ある軸が丸ごと抜けたりする
- 空欄は許容せず、「検討した上でテストしない」は `-` や注記で明示する

### 定期的にレビューする

- カバレッジマップは、テスト計画時とリリース前にレビューする
- レビュー観点:
  - 新機能に対応する行が追加されているか
  - カバーしている層は妥当か
  - 空欄が残っていないか
  - 特定の層に偏り過ぎていないか
  - 委譲の理由は妥当か（環境変化により、自動化可能になっている場合もある）

## 自動テストの維持の心得

**_自動テストスイートの信頼性を継続的に維持する。_**

### テストコードの棚卸し

- 仕様変更・機能削除に伴い不要になったテストは、プロダクションコードと同様に削除する
  - 死んだテストを放置すると、テストスイートがノイズまみれになり、何を保護しているかが見えなくなる
  - 「テストが増え続けること」は良いことではない - 現在の仕様を正確に表現しているテストだけが資産
- 棚卸しのタイミング
  - 機能削除時: 対応するテストをセットで削除する（削除し忘れると後からパスし続ける幽霊テストになる）
  - 大規模リファクタリング時: テストの意図とプロダクションの構造のずれを整理する
  - カバレッジマップの定期レビュー時: 観点が現在の仕様と合致しているかを確認する
- テストを削除することへの心理的抵抗を持たない
  - 「消すと何かが壊れそう」と感じる場合は、そのテストが守っている観点を別の層で担保してから消す
  - テストの存在価値は「本数」ではなく「仕様への対応関係」にある

### CI 実行時間の管理

- テストスイートが成長するにつれ CI が遅くなり、フィードバックループが壊れる
  - CI が遅くなると「とりあえず Push してCIに任せる」習慣が生まれ、ローカル実行が疎かになる
  - CI が壊れたままマージされることが増え始めたら、実行時間の見直しサイン
- 実行時間を層ごとに計測し、ボトルネックを特定する
  - 遅い原因の多くは「本来下層で担保できる観点を上層で検証している」こと - 層割り当ての見直しが根本対処になる
  - 特定のテストだけが突出して遅い場合は、外部依存のスタブ化や並列実行を検討する
- 層ごとに実行タイミングを分離し、PR ごとに全層を回さない設計にする
  - 例: 単体テストは PR 時に全実行、統合テスト以上は main マージ時または夜間バッチ
  - ただし「CIで動かさない層」はローカル実行の習慣とセットにしないと、問題の検知が遅れる
- CI 実行時間を定期的にモニタリングし、閾値を設ける
  - 例: 単体テストが 3 分を超えたら要調査、10 分を超えたら並列化または層の見直しを義務とする

### フレーキーテストへの対処

- フレーキーテスト（実行のたびに成否が変わる不安定なテスト）を放置すると、テストスイート全体への信頼が失われる
  - 「失敗してもどうせフレーキーだろう」という空気が生まれ、本当の失敗を見逃すようになる
- 発生したら即対処する - 後回しにすると蓄積してCIが形骸化する
  - テスト分離の破綻（実行順依存、共有状態の汚染）が最も多い原因 → テストデータのリセット・ロールバックを徹底する
  - 非決定的な値（時刻・乱数）を制御せずに使っている → モックまたは固定値に置き換える
  - 外部サービスへの実通信を含む → タイムアウト設定の見直し、またはスタブへの切り替えを検討する
- 即時修正が難しい場合は「隔離（quarantine）」する
  - CIのブロック対象から一時的に外し、隔離リストとして別管理する
  - 隔離したまま放置しない - 隔離は修正するまでの猶予であり、恒久的な回避策ではない
- カバレッジマップの定期レビュー時に、隔離中のテストがないかを確認する観点を加える

## 手動テスト運用の心得

**_テスト項目チェックリストのクラスとインスタンスを使い分ける。_**

### 仕様書（クラス）と実施記録（インスタンス）を分離する

- テスト仕様書: 「何を検証するか」のチェックリスト定義 - リポジトリで管理し、PR レビュー対象にする
- テスト実施記録: 「いつ・誰が・どんな結果だったか」の記録 - Issue やテスト管理ツールで管理する
- 仕様書に実施結果を書き込むと、差分管理が煩雑になり、仕様変更の PR レビューが困難になる

### テストケースのフォーマットを統一する

- 「操作」「期待結果」「前提条件」を明示したフォーマットに統一することで、テスト実施者が迷わない
  - 参考: [テスト仕様書の構成例](./documentations-cheatsheet.md)
- フォーマットが統一されていないと、実施者によって確認の深さや記録の粒度に差が生まれる

### テスト証跡の記録の質を高める

- 手動テストの内訳を「作業（オペレーション）」と「判断」に分解する
  - 作業: コマンド実行、手順書通りの進行、結果の記録
  - 判断: テスト結果の目視確認、異常の検知、妥当か否かの判定
- 作業部分は、スクリプト化や AIエージェントへの委任等により自動化できないか検討する
- 実施中のコマンド結果・スクリーンショット等は随時記録する（Issue コメント等）
  - 証跡は「テストが正しく実施されたこと」の証明だけでなく、障害発生時の調査や、テスト改善のヒントとしても使える
  - 粒度の目安: 後から同じ手順を再現できる程度
- NGとして検知したものは、整理して一覧化する
- 作業と判断の分解は、テストを手動で繰り返す中で見えてくることもあるため、細やかに記録しておくと資産になる

### 探索的テストを組み合わせる

- 仕様書に基づくテストだけでは発見しにくい問題を探索的テストで補う
  - 例: 仕様の隙間（「ユーザーが想定外の操作をした場合」）を担当者の経験で探る
  - 例: 新機能リリース前の「なんとなく気になる動作」の確認
- 探索的テストで発見したバグや観点は、テスト仕様書にフィードバックして資産化する

## テストデータ管理の心得

**_テストデータの配置・取得手順を明確にし、再現可能性を担保する。_**

### テストデータを「誰でも・いつでも・同じ状態で」用意できるようにする

- テストデータの作成手順をコード化し、手動セットアップを排除する
  - 例: DB のシードスクリプトをリポジトリに含め、`make seed-test` 一発で再現できるようにする
  - 例: フィクスチャファイル（JSON/YAML）で定義し、テスト実行前に自動投入する
- テスト間でデータが汚染されないように設計する
  - 例: 各テストのセットアップ・ティアダウン（setUp/tearDown）でデータをリセットする
  - 例: テストごとにトランザクションロールバックを利用する

### テストデータは「テストの意図」を表すように設計する

- テスト対象の観点に関係するデータのみを最小限用意し、無関係なデータで意図が埋もれないようにする
  - 例: 権限チェックのテストなら「ロール属性」だけを設定し、氏名・住所は任意値でよい
- 本番データをマスキングして使うことは、データ構造の再現性は高いが注意が必要
  - 本番データは想定外の値を含む可能性があり、テストの意図が不明確になる
  - マスキング漏れによる個人情報の混入リスクがある

### テストデータの独立性

- テスト間でデータを共有しない - 各テストが独立してセットアップ・クリーンナップする
  - テスト実行順序への依存が発生すると、CI での並列実行やローカルでの単独実行が困難になるリスクがある
- 大量データが必要なテスト（ページネーション等）は、テスト内でシードデータを生成する
  - 固定のシードデータベースに依存すると、スキーマ変更時に全テストが壊れるリスクがある

### テストデータの腐敗を防ぐ

- テストデータも仕様変更のたびに更新し、プロダクションコードと同期させる
  - 「古いフィクスチャが原因でテストが通らなくなった」を防ぐ
- 大量のフィクスチャを手で管理するのは限界があるため、ファクトリパターン（Factory Boy 等）を活用する
  - 例: `UserFactory(role="admin")` のように、テストごとにカスタマイズしたデータを動的生成する
