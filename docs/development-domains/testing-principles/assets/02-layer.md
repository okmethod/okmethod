# layer: どのようにテスト層の境界と委譲を設計するか

**コンセプト**: 各観点をコスト効率の最も良い層に割り当て、層の間の重複と漏れを防ぐ

---

## 関連キーワード

- **Test Double** (テストダブル - モック・スタブ・フェイクの総称)
- **Mock** (モック - 関数が呼び出されたか・何回呼ばれたかの検証を目的とした代替)
- **Stub** (スタブ - 固定の戻り値を返す代替。呼び出し検証はしない)
- **Fake** (フェイク - 本物に近い動作をする軽量な代替。インメモリ DB など)
- **Spy** (スパイ - 本物の挙動を維持しつつ呼び出し履歴を記録する代替)
- **Contract Test** (コントラクトテスト - コンシューマとプロバイダ間のインターフェース境界をE2Eと独立して検証する手法)
- **Test Isolation** (テスト分離 - テスト間で状態が干渉しない設計)
- **Over-Mocking** (オーバーモッキング - モックの過剰使用による信頼性の低下)

---

## 層割り当ての判断基準の心得

**_外部依存・再現コスト・検証の信頼性から、最適な層を選ぶ。_**

### テストを書く前に「この観点はどの層が最適か」を問う

- 外部依存の有無と、依存先の種類で層を決める（参考:[層割り当てチートシート](./layer-assignment-cheatsheet.md)）
  - 外部依存がない → 単体テスト
  - DB に依存する → DB 結合テスト
  - 自チームの API に依存する → API 結合テスト
  - 外部サービスに依存する → 代替可能なら スタブ結合テスト、代替不可なら API結合テスト または E2E テスト
- 誤った層に割り当てると、コストが余分にかかるか信頼性が下がるかのどちらかが起きる
  - 例: 四則演算のロジックテストを E2E で書く → 低速・不安定になるだけで信頼性に貢献しない
  - 例: 外部 API との通信をすべて単体テストでモックする → 実際の統合の問題を見逃す
- 同じ観点を検証できる層が複数ある場合、原則として最も低コストに検証できる層（≒下層）を選ぶ
  - 例: バリデーション → 単体テスト、DBのフィルタ・ソート → DB結合テスト、外部LLM連携のエラーハンドリング → 単体テスト（モック）+E2Eテスト
  - 上層のテストは、検証済みであることを前提に、結合部分だけを検証する

### 各層で適切な抽象レベルで検証する

- テスト層が上になるほど振る舞い（ユーザー・システム視点の入出力）に、下になるほど実装の詳細に踏み込む
  - 上層（E2E・結合）: ユーザー操作やシステム間連携の結果など、観察可能な振る舞いを検証する
  - 下層（単体）: バリデーションの境界値・分岐条件・エラーパスなど、実装を知ることで洗い出せる観点を、公開インターフェース経由で検証する — これは自然であり、むしろ必要
- 層に不釣り合いな実装詳細をテストすると、脆いテストになってしまうため注意する
  - 悪い例（上層で実装詳細）: E2E で内部メソッドの呼び出し順序を検証する → リファクタリングのたびに E2E が壊れる
  - 悪い例（下層で詳細すぎる）: プライベートメソッドを呼び出し検証している → メソッド名変更でテスト失敗
  - 悪い例（層に関わらず）: スタブへのアサーション → 入力の制御手段を検証対象にしてしまっている
- リファクタリング耐性も層によって異なる
  - 上層: 振る舞いが変わらなければテストが壊れないことが理想
  - 下層: 仕様変更（新しい境界値の追加等）に伴うテスト変更は自然

### 「テストできないコード」はプロダクション設計の問題と捉える

- テストが書けない場合、まずプロダクションコードのリファクタリングを検討する
  - 外部依存をコンストラクタや DI で注入するようにする
    - 依存をハードコードするとテスト時に差し替えられないが、DI にするとモック・フェイクへの置き換えが可能になる
  - 副作用のある処理と純粋な計算ロジックを分離する
    - 副作用（DB 読み込み・API 呼び出し等）と計算ロジックが混在していると、計算ロジックだけを単体テストできなくなる
- 「テストのためのリファクタリング」はしばしばコードの設計改善そのものになる
- リファクタリングが困難な場合（レガシーコード・外部ライブラリの制約等）は、リスクとして管理する

### テスト分離（Test Isolation）を設計する

- テスト間で状態が干渉しないよう設計する
  - 各テストは独立して実行でき、実行順序に依存しない
  - 共有状態（DB レコード、ファイル、グローバル変数）はテスト毎にリセットする
  - テスト分離が崩れると、実行順で成否が変わるフレーキーテストの原因になる
- DB 結合テストでの主な分離手法
  - テスト用 DB は本番と同じエンジンを使う: SQLite 等で代替すると、本番 DB 固有の挙動（型の厳格さ・関数・制約等）との乖離リスクがある
  - トランザクションロールバックする: テスト終了時にロールバックして DB 状態を元に戻す
  - テストごとにデータをリセットする: テスト固有のデータを使い、他テストと干渉しないようにする
- API 結合テストでの主な分離手法
  - テスト専用の認証トークン・ユーザーを使う: 認証をバイパスせず、テスト用クレデンシャルを用意する - スキップすると認可ロジックがテストの射程に入らない
  - テストデータのセットアップ・クリーンアップ: テスト開始前にデータを作成し、終了後に削除する（または DB レベルのロールバックに委ねる）

## テストダブル・実物の使い分けの心得

**_テストダブルを有効に使える場面と、実物でしか検証できない場面を見極める。_**

### テストダブルの選択優先順位

- **実物 → フェイク → スタブ → モック** の順で選ぶ
  - 実物（Real）: テストで許容できる速度・コストで動くなら常に優先する - 代替物を作ること自体がコスト
  - フェイク（Fake）: 実物が重い場合の次善策。本物に近い動作を持つため、スタブやモックより信頼性が高い（例: インメモリ DB）
  - スタブ（Stub）: 固定値を返せば十分で、呼び出し検証が不要な場合に使う
  - モック（Mock）: 副作用（外部命令）の発生有無・回数を検証する必要がある場合のみ使う - 実装詳細への結合を生むため多用しない

### テストダブルが有効な場面

- ロジックの分岐網羅が目的の場合 - 外部依存の応答パターンを制御しやすい
  - 例: 外部APIが異常レスポンス(200/400/500 等)を返した場合のハンドリングロジック
  - 例: タイムアウト・接続エラー時等のリトライロジック
- 外部サービスへの通信などの副作用がある場合
  - 例: 利用料金がかかる、外部サービスがステートを持っている
  - 「完全モック（単体テスト）」か「実環境（E2E）」かの二択ではなく、スタブで外部サービスを差し替えつつ自チームのコードを実際に動かす中間選択肢（スタブ結合テスト）もある
- 異常系・エッジケースなどの再現コストが高い場合
  - 例: 外部サービスの一時停止、レート制限
  - 例: ディスク容量超過、メモリビジー、ネットワーク遅延
- 非決定的な値に依存する場合
  - 例: 時刻、乱数
- 外部依存先が未提供の場合
  - 例: 依存している外部サービスを別チームが並行で開発している
- DI（依存性注入）を活用し、外部依存物を交換しやすくしておくと、安全に切り替えられる
  - 例: 初期はモックでテスト実装し、成長してきたらインメモリDBに交換する
  - DI で交換可能な境界が、そのままテスト層の分離点にもなる

### テストダブルでは不十分な場面

- 実際のデータ形式・プロトコルの互換性が検証対象の場合
  - DB のクエリそのものの正確性を確認したい場合
  - ORM/クエリビルダのマッピング動作を確認したい場合
  - メッセージキューのエンキュー・デキュー動作を確認したい場合
- サービス間のスキーマ・型・フィールドの整合性を確認したい場合
  - 複数チームにまたがる、またはE2Eが肥大化している場合はコントラクトテストとしてE2Eから切り出すと有効
  - コンシューマ（呼び出し側）とプロバイダ（提供側）が独立して検証できるため、E2Eを動かさなくてもI/F合意の整合性を担保できる
- パフォーマンス特性が検証対象の場合
- テストダブルによる実物と同じ振る舞いの提供、または保証が困難な場合

### テストダブルの過剰使用（Over-Mocking）に注意する

- テストがモックだらけになると「動作しているのはモックのシナリオであり、実際の統合ではない」という罠に陥る
  - 例: モックでテストをパスしたのに、本番連携でバグが発生する
- プロダクションコードが成長すると、それを動かすためのモックも増える
  - 例: カラム追加でモック修正、クエリ変更でモック修正
  - モックの保守コストがテスト本来の目的を圧迫し始めたら、戦略の転換点
- スタブへの呼び出し検証はアンチパターン
  - スタブは「戻り値を制御する入力の代替」であり、副作用を検証する対象ではない
  - スタブへのアサーションは実装詳細への結合を生み、リファクタリングで壊れやすいテストになる
  - 呼び出し自体を検証する必要がある場合はモック（Mock）を使う

## 委譲の記録の心得

**_ある層で検証しないと決めた観点は、理由とともにテスト仕様書に明記する。_**

### 委譲は設計判断である

- 理由を書かずにテストを省略すると、別担当者が「漏れ」と「意図的な省略」を区別できない
  - 「漏れ」と誤解してテストを重複追加したり、「意図的な省略」と誤解して漏れを見逃したりする恐れがある
  - 悪い例: カバレッジマップが空欄（漏れなのか意図なのか区別不可）
  - 良い例: カバレッジマップにハイフンを書いた上で、「外部 API のレスポンス解析はスタブ結合テストで担保済みのため、E2E では再検証しない」と注記がある
- 委譲先を明記し、その判断を検証できるようにする
  - 「この観点は単体テストに委譲」「この観点は手動テストに委譲」のように、どこで担保されるかをセットで書く

### 「テストしない」ことにも品質がある

- 委譲先が明確 - 意図的な省略は漏れでなく、効果的なテスト設計といえる
- リスク受容可能と判断 - リスク（発生確率・影響範囲・影響度）を検討済みであれば問題なし
- リスクとして認識 - 認識しているだけで、リスクを正しく見積もれていない状態では高リスク
- 単なる漏れ - 認識できていない状態が最も高リスク
