# ドキュメンテーションチートシート

**_前提：システムの規模に応じて、記載粒度や書くこと・書かないことのバランスを見極めながら書く_**

---

## テスト戦略書 構成例

「品質をどう担保するか」の基本方針文書。開発開始前に定め、ビジネス・エンジニア双方が参照する。

```
1. 目的・スコープ
   - 対象プロダクト・バージョン
   - 対象外スコープ（明示的に除外するもの）
2. 品質目標とリスク優先度
   - リリース可否の受け入れ基準
   - リスク優先度（要件の Must/Should/Could に対応させる）
3. テスト層の定義
   - 各層の名称・責務・実行環境
   - 例: 層1=BE単体（ローカル）、層2=BE-DB結合（CI）、層3=E2E（STG）
4. 自動化方針
   - 自動化する層・しない層の判断根拠
   - CI/CD への組み込み方針（実行タイミング・マージブロック条件）
5. テストできない領域と対策
   - 意図的に対象外とする観点と、その受容理由
```

## テスト計画書 構成例

「何を・いつ・誰が」を非エンジニアとも合意する文書。  
プロジェクト計画書に含めることが多い。  
技術詳細は含めないよう注意する。

```
1. テスト対象・スコープ
   - 対象機能一覧（要件定義の優先度と対応させる）
   - 対象外の機能・観点
2. スケジュール
   - 何ショット行うか
     例: 1st=問題の洗い出し、2nd=バグ修正確認 + 追加改修分、3rd=最終確認
   - テスト開始予定・終了予定
   - バッファ（最終ショット後に再テストを回す余地）
3. 体制・役割分担
   - テスト実施者・レビュワー・リリース判断者
4. 必要環境・ツール
   - テスト環境（DEV, STG 等）、テスト管理ツール
5. 完了基準
   - すべての必須テストが PASS していること
   - 未解決の重大バグが 0 件であること
```

## テスト設計書 構成例

「どの観点を・どの層で・どう検証するか」を定める技術文書。エンジニアが作成・維持する。

```
1. 対象機能・コンポーネント
2. 検証観点一覧
   - 機能仕様: 正常系 / 異常系 / 境界値
   - 非機能: 性能 / セキュリティ / 互換性
   - 運用: デプロイ後の疎通確認 / リグレッション
3. カバレッジマップ（→ テンプレート参照）
4. テストダブル方針
   - モック・スタブ・フェイクの使い分けルール
5. 委譲の記録
   - 「この観点はどの層に委譲し、なぜその層で担保できるか」を観点ごとに明記
```

## カバレッジマップ テンプレート

テスト設計書に含める。行=検証観点、列=テスト層。  
全てを1つのマップにしようとせず、検証観点に基づきうまく分離するとよい。

| #   | 検証観点                   | 層1<br>単体 | 層2<br>DB結合 | 層3<br>API結合 | 層4<br>E2E | 備考                                 |
| :-- | :------------------------- | :---------- | :------------ | :------------- | :--------- | :----------------------------------- |
| 1   | 入力バリデーション         | ✔︎           | -             | -              | ✔︎          |                                      |
| 2   | DB への保存と取得          | -           | ✔︎             | -              | -          |                                      |
| 3   | 外部API エラーハンドリング | ✔︎(スタブ)   | -             | -              | -          | 実サービスでの挙動は手動テストに委譲 |
| 4   | 認証・認可                 | -           | -             | ✔︎              | -          |                                      |
| 5   | 大容量データ処理           | -           | -             | -              | -          | 性能テストフェーズに委譲             |

**セルの記号**

- `✔︎` : この層でカバーする
- `-` : 意図的に対象外（別層に委譲済み、またはリスク受容済み）
- `?` : 検討中（仮置きとしては許容、レビュー前に解消すること）
- 空欄: 使用しない
- 適宜、注記を含む（例: ✔︎(スタブ) ）

## テスト仕様書（テスト項目書）構成例

具体的な操作・入力・期待結果を記述する。手動テストのチェックリストとして使用する。

```
## {機能名 / シナリオ名}

**前提条件**
- ログイン済みであること
- {必要なデータが存在すること}

### 正常系
- [ ] {操作} → {期待結果}
- [ ] {操作} → {期待結果}

### 異常系
- [ ] {操作} → {期待結果}

### 境界値
- [ ] {操作} → {期待結果}
```

**記述ルール**

- 操作と期待結果を同一行に書く（`{操作} → {期待結果}` の形式）
- 前提条件（ログイン済み、特定データが存在する等）はセクション冒頭にまとめる
- 実施記録（日付・担当者・結果）はテスト仕様書に書かず、Issue やテスト管理ツールに残す
