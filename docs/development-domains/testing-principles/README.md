# テスト設計の心得

## 大原則： 表裏一体のプロダクションとテストを共進化させる！

- プロダクション層の分離設計がテスト層を生み、テストの限界がプロダクション設計を変える - この反復がアーキテクチャを成熟させる
- テスト設計はプロダクション設計と同時に育てる - 先に設計するのでも、後から追加するのでもない
- 「何をテストするか」と同じくらい、「何をテストしないか」の判断も重要
- テストもコードである - 読みやすさ・保守しやすさはプロダクションコードと同じ重要度で考える

---

## 前提: 本ガイドにおける用語の定義について

ソフトウェア開発において「テスト設計」「テスト仕様」などの言葉は、組織やプロジェクトによって指す範囲が大きく異なる。  
本ガイドでは、「抽象から具体への粒度」に基づいて、以下の 4 つの概念に分類してまとめている。

- `テスト戦略`
  - 問い: **なぜ・どういう方針で**テストするか
  - 役割: テスト全体の基本方針を定める。品質目標・リスクの優先順位・テストアプローチを決める。
  - 粒度: 組織/プロジェクト全体
- `テスト計画`
  - 問い: **いつ・誰が・何を・どのリソースで**テストするか
  - 役割: 戦略を実行計画に落とす。スコープ・スケジュール・体制・環境を具体化する。
  - 粒度: プロジェクト
- `テスト設計`
  - 問い: **どの観点で・どのように**テストするか
  - 役割: テストすべき条件・観点を洗い出し、テスト層への構成・検証方法を決める。
  - 粒度: 機能/コンポーネント/ツール
- `テスト仕様`
  - 問い: **何を**テストするか
  - 役割: 具体的なテストケース（入力・手順・期待結果）を記述する。
  - 粒度: テストケース

なお、戦略→計画→設計 は「抽象→具体」への粒度変化だが、設計と仕様は別軸の問いを担う。  
テスト設計が「枠組み（層・技法・ツール）」を決め、テスト仕様が「中身（ケース・入力・期待結果）」を記述する。

## [approach: どのような戦略でテストの全体像を計画するか](./assets/01-approach.md)

コンセプト: **_テスト全体像を定め、プロダクト構造に合わせてテスト層と自動化戦略を設計する_**

トピック:

- テスト戦略とテスト計画: 「何を作るか」と「どう検証するか」をセットで考える
- テスト層の定義: プロダクトのコンポーネント構成に基づいて、テストピラミッドの層を設計する
- 自動化方針の決定: 各層の自動化レベルと実行環境を決め、CI/CD に組み込む
- 仕様書とコードの棲み分け: テスト仕様書は「観点の網羅性」、テストコードは「ケース詳細」を担う

## [layer: どのようにテスト層の境界と委譲を設計するか](./assets/02-layer.md)

コンセプト: **_各観点をコスト効率の最も良い層に割り当て、層の間の重複と漏れを防ぐ_**

トピック:

- 層割り当ての判断基準: 外部依存・再現コスト・検証の信頼性から、最適な層を選ぶ
- テストダブル・実物の使い分け: モック等を有効に使える場面と、実物でしか検証できない場面を見極める
- 委譲の記録: ある層で検証しないと決めた観点は、理由とともにテスト仕様書に明記する

## [verification: どのように検証フローを運用するか](./assets/03-verification.md)

コンセプト: **_テスト仕様書を「レビュー可能な観点一覧」として維持し、テスト実施の運用フローを確立する_**

トピック:

- カバレッジマップの設計: テスト層 × 検証観点のマトリクスで、網羅性を俯瞰する
- 自動テストの維持: 自動テストスイートの信頼性を継続的に維持する
- 手動テストの運用: テスト項目チェックリストのクラスとインスタンスを使い分ける
- テストデータの管理: テストデータの配置・取得手順を明確にし、再現可能性を担保する
