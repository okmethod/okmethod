# 要件定義の心得

## 目的

要件定義・要求分析・業務要件/システム要件の意思決定をサポートする。  
「要求は言語化されるまで誰も本当の姿を知らない」を前提に、**対話と反復によって本当に必要なものをあぶりだし、Why/What/How のトレーサビリティを確保する**ことを支援する。

## アセット

このスキルのナレッジファイルは `assets/` に配置されている:

- **`assets/01-requirement.md`** — 要求収集・コンセプト策定・スコープ定義
- **`assets/02-business.md`** — ユビキタス言語・アクターモデリング・業務フロー
- **`assets/03-specification.md`** — システム要件（機能・非機能・インターフェース・データ）
- **`assets/documentations-cheatsheet.md`** — 要件ドキュメンテーションチートシート
- **`assets/non-functional-requirements-overview.md`** — 非機能要件カテゴリ概要

---

## 発動タイミング（TRIGGER）

以下のキーワード・文脈が含まれる場合に発動する:

- 要件定義・要求分析・要件整理の相談・レビュー
- ユーザーストーリー・ユースケース・受け入れ条件の作成
- 業務フロー・業務モデル・ドメイン知識の整理
- ユビキタス言語・用語定義・アクターモデリングの相談
- スコープ定義・MVP 設計・優先度付けの判断
- 機能要件・非機能要件の定義・分類
- 要件と設計の境界に関する相談

## スキップ条件（SKIP）

以下の場合はこのスキルを使用しない:

- アーキテクチャ設計・技術スタック選定（→ アーキテクチャ設計の心得を参照）
- テスト設計・テスト戦略（→ テスト設計の心得を参照）
- コーディング実装・リファクタリングの具体的な方法（→ コーディングの心得を参照）
- 特定フレームワーク・ライブラリの API の使い方

## 自律実行ポリシー

- **コンテキストから推論できる場合は自律判断する** — ユーザーへの確認は最小限にとどめる
- **確認が必要な場合** — 回答の方向性が変わる制約条件が不明な場合のみ質問する
- **質問はまとめて 1 ターンで行う** — 必要な確認事項は一度にまとめ、往復を最小化する（最大 4 問）

---

## 実行手順

### Step 1: 問いを分析する

コンテキストから自律推論する。推論できない場合のみユーザーに確認する（1 ターンにつき最大 4 問）:

1. **対象スコープ**: プロジェクト全体 / 単一機能 / 単一業務領域（コード・文脈から推論）
2. **問いの階層**（キーワードから自律推論。判断が難しい場合は独自用語（Requirement/Business/Specification）を使わず確認する。例：「全体のスコープ・方向性についてですか？業務の詳細についてですか？システム仕様についてですか？」）:
   - **Requirement** — 要求収集・コンセプト策定・スコープ定義。変更コスト: 特大。キーワード: ユーザー課題、ビジネス価値、MVP、スコープ、Why
   - **Business** — 業務要件定義・業務モデリング。変更コスト: 大。キーワード: 業務フロー、アクター、ペルソナ、ユビキタス言語、ドメインモデル
   - **Specification** — システム要件定義。変更コスト: 中〜大。キーワード: 機能要件、非機能要件、インターフェース、データモデル、制約条件
3. **制約条件**: ステークホルダー構成・チーム規模・フェーズ（不明な場合のデフォルト前提: スタートアップ初期〜中期の Web サービス）

### Step 2: 対応アセットを読み込み、関連セクションを特定する

問いの階層に応じたファイルを **Read ツールで** 読み込む。複数の階層にまたがる場合は、**変更コストの大きい上位階層から順に**（Requirement → Business → Specification）読み込む:

| 問いの階層 | 読み込むファイル |
|---|---|
| Requirement（要求収集・コンセプト・スコープ） | `assets/01-requirement.md` |
| Business（業務要件・アクター・業務フロー） | `assets/02-business.md` |
| Specification（システム要件・機能・非機能） | `assets/03-specification.md` |

チートシートは以下の条件で参照する:

- 要件ドキュメントの構成・種類の相談が出た場合 → `assets/documentations-cheatsheet.md`
- 非機能要件のカテゴリ分類・網羅性の確認を求められた場合 → `assets/non-functional-requirements-overview.md`

**読み込んだ後の処理**:

1. ユーザーの問いと照合し、関連するセクション・トピックを特定する
2. 関係ないセクションは読み飛ばし、関連箇所のみ抽出する
3. 抽出した内容を Step 3〜4 の回答根拠として使う

### Step 3: 関連トピックを提示する

**問いが 1 つのトピックに絞られている場合は Step 3 をスキップして直接 Step 4 へ進む。**

複数トピックにまたがる場合のみ、アセット内から関連するトピックを **2〜4 件** ピックアップして提示する。  
選定基準: **① 直接関連するもの優先、② 変更コストの大きい階層（Requirement > Business > Specification）を先に、③ 最大 4 件に絞る**。  
ユーザーが深掘りしたいトピックを選べるようにすることが目的。

**トピックマップ（変更コストの大きい順に並べている）**:

| キーワード例 | 参照トピック | ファイル |
|---|---|---|
| ユーザー課題 / 問題発見 / 洞察 | 洞察と目標設定の心得 | `01-requirement.md` |
| ビジネス価値 / コンセプト / ストーリー | コンセプト策定の心得 | `01-requirement.md` |
| スコープ / MVP / やらないこと | スコープ定義の心得 | `01-requirement.md` |
| 用語 / ユビキタス言語 / 言葉の定義 | ユビキタス言語定義の心得 | `02-business.md` |
| アクター / ペルソナ / ユーザー像 | アクターとペルソナのモデル化 | `02-business.md` |
| ドメインモデル / ナレッジ / 概念 | ドメイン知識の構造化 | `02-business.md` |
| 業務フロー / 例外 / シナリオ | 業務フローと例外設計 | `02-business.md` |
| 制約条件 / 前提条件 / コントロール不能 | 前提条件と制約条件の整理 | `03-specification.md` |
| コンポーネント / システム境界 / 責任範囲 | システム方式と境界設計 | `03-specification.md` |
| データ要件 / 論理データモデル / エンティティ | データ要件定義 | `03-specification.md` |
| 機能要件 / ユースケース / 処理ロジック | 機能要件定義 | `03-specification.md` |
| インターフェース / API / UI / 外部連携 | インターフェース要件定義 | `03-specification.md` |
| 非機能要件 / パフォーマンス / セキュリティ / SLA | 非機能要件定義 | `03-specification.md` |
| 要件書 / ドキュメント / 仕様書 | 要件ドキュメント管理の心得 | `03-specification.md` |

### Step 4: トレードオフを列挙して推奨を出す

- 選択肢を 2〜3 個挙げる
- 各選択肢の「得るもの」と「失うもの」を明示する
- 「最善解」ではなく「**最悪回避**」の観点で推奨を出す

### Step 5: 意思決定を記録する

以下のいずれかに該当する場合に要件サマリを生成する:

- ユーザーが明示的に記録作成を依頼した場合
- Step 4 で Requirement / Business 階層など、後戻りコストの大きい決定について、ユーザーが受諾・確定を示す発言をした場合

「なぜこのスコープ・要件にしたか」と「何を諦めたか（対象外にしたか）」をセットで残す。

## アウトプット形式

### 要件定義提案

```
## [テーマ]

### 選択肢

| 選択肢 | 得るもの | 失うもの |
|---|---|---|
| A | ... | ... |
| B | ... | ... |

### 推奨

**[推奨案]** を推奨する。理由: [根拠]

### 諦めること

[この選択で手放すもの]

### 関連トピック

詳しく掘り下げたい場合は以下を参照:
- [トピック名] → `ファイル名`
```

### 要件レビュー

```
## レビュー結果

### 問題点

- [どの原則が守られていないか・具体的な箇所と理由]

### 代替案

[より適切な定義と根拠]

### 関連トピック

- [トピック名] → `ファイル名`
```

### 要件サマリ

```
## 要件サマリ

- **Status**: Accepted / Proposed / Deprecated
- **Context**: [背景・制約・ステークホルダー]
- **Decision**: [なぜこのスコープ・要件にしたか・根拠]
- **Scope**: [対象範囲と対象外]
- **Consequences**: 得るもの: [得るもの] / 諦めること: [諦めること]
```

## 設計判断の軸

1. **Why を先に固める** — 目的・課題が曖昧なまま What/How に進まない
2. **変更コストの大きい決定から先に** — Requirement > Business > Specification の順で整理する
3. **最悪回避** — 最善解より「後悔しにくい選択」を優先する
4. **トレーサビリティを記録する** — Why/What/How の繋がりを文書に残し、後工程での変更コストを下げる
