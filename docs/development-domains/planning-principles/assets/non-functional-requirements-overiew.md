# 非機能要求グレードまとめ

## 非機能要求グレードとは

非機能要求グレードとは、 IPA（情報処理推進機構）が「非機能要求」についてユーザと開発者との間での共通理解形成を目的とし、
非機能要求項目を網羅的にリストアップして分類するとともに、それぞれの要求レベルと各項目の定義のマッピングを示したツール群。

非機能要件を考える or 決める際に、どのような項目を定めるべきか、
また、ある項目をどのようなレベル感として設定するべきか、の候補として参考になる。

参考: [システム構築の上流工程強化（非機能要求グレード）紹介ページ](https://www.ipa.go.jp/archive/digital/iot-en-ci/jyouryuu/hikinou/ent03-b.html)

---

## 非機能要件の 6 大カテゴリ

1. **可用性(Availability)** : システムサービスを継続的に利用可能とするための要求
   キーワード：運用スケジュール、稼働率、RTO/RPO、冗長化、バックアップ、災害対策

2. **性能・拡張性(Performance)** : システムの性能、および将来のシステム拡張に関する要求
   キーワード：応答時間、同時アクセス数、リソース使用率、サイジング、キャパシティプランニング

3. **運用・保守性(Maintainability)** : システムの運用と保守のサービスに関する要求
   キーワード：監視、計画停止、ログ保存期間、バックアップ運用、障害検知、障害対応、運用体制

4. **移行性(Portability)** : 現行システム資産の移行に関する要求
   キーワード：移行期間、移行データ量、移行スケジュール、並行稼働

5. **セキュリティ(Security)** : 情報システムの安全性の確保に関する要求
   キーワード：認証・認可、暗号化、脆弱性対策、不正検知、監査

6. **システム環境(Environment)** : システムの設置環境やエコロジーに関する要求
   キーワード：耐震、電源設備、空調、エコロジー(CO2)
   （クラウド利用時は考慮な場合も多い）

## 非機能要求のグレード分類

非機能要求グレードでは、下記 3 つのモデルシステムを定義している。  
また、各モデルシステムについて、各非機能要求項目のレベル感を定めている。

- 社会的影響が殆ど無いシステム
- 社会的影響が限定されるシステム
- 社会的影響が極めて大きいシステム

これ自体も参考になるが、このまま使うにはやや抽象度が高い。  
そのため、これを参考に身近な開発案件の規模ごとのレベル感を掴んでおいたり、事前に各項目のレベルを定義しておくと良い。

1. **ミニマム** : 止まってもなんとかなる。障害対応はベストエフォート。
   (例) 社内限定の小規模ツール、検証用(PoC)、一時的なキャンペーンサイト

2. **標準** : 日中業務時間は稼働していないと困る。障害対応は営業時間のみで、xx 営業日以内に復旧など。
   (例) 一般的な社内業務システム、情報系 Web サイト

3. **高信頼** : 原則 24 時間 365 日稼働（計画停止可能）。障害対応は夜間・休日の運用体制が必要で、xx 時間以内に復旧など。
   (例) 一般的な BtoC EC サイト、顧客管理基幹システム(CRM)

4. **ミッションクリティカル** : 数分〜数秒の停止も許されない。多重化・自動切替が必須。
   (例) 金融機関の勘定系、公共インフラ、大規模プラットフォーム
