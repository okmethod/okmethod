# 要件定義の心得

## 大原則： 本当に必要なものを、対話と反復で考え抜く！

- 要求は言語化されるまで誰も本当の姿を知らない - 対話と反復によって初めて輪郭が現れる
- 目的(Why) と手段(What/How) を分けて考え、まずは目的を深掘りして認識を揃える
- ある工程と前工程のトレーサビリティを確保する
- 悩んだら、前工程=目的に立ち戻って考える

---

## 前提： 本ガイドにおける用語の定義について

要件定義を取り巻く用語（要求・要件・業務要件・システム要件など）は、組織やプロジェクトによって使われ方が大きく異なる。  
本ガイドでは、**「Why/What/How の入れ子構造」** という軸でこれらを整理する。

各工程の「How（どのように解決するか）」が、次工程の「What（何を解決するか）」になる - この連鎖とトレーサビリティがこの構造の核心であり、「悩んだら前工程に立ち戻る」という大原則の行動指針もここから導かれる。

| 用語           | 担う問い        | 主体             | 位置づけ                                                   |
| -------------- | --------------- | ---------------- | ---------------------------------------------------------- |
| `要求`         | Why / What      | 事業側（発注者） | 達成したい目的・課題。抽象度が高いインプット               |
| `業務要件`     | How → 次の What | 事業・技術双方   | 要求の How。業務レベルの解決策                             |
| `システム要件` | How → 次の What | 技術側（受注者） | 業務要件の How。システムの外部仕様                         |
| （`設計`）     | How             | 技術側           | システム要件の How。システムの内部仕様（本ガイドの対象外） |

ただし、要件と設計は不可分であり、What を説明するために How に踏み込まざるを得ないこともある。  
この点に留意しつつ、本ガイドでは「要件」側に軸足を置いて説明する。

**参考**: [ドキュメンテーションチートシート](./assets/documentations-cheatsheet.md)

### 補足: `要求` と `要件` の使い分けについて

「IPA 共通フレーム 2013」において、下記のプロセスは明確に定義されている。

- `企画プロセス`: 経営戦略や事業計画に基づいて、システム化の目的、目標、範囲（スコープ）、およびその実現の可能性を決定するプロセス。
- `要件定義プロセス`: システムが満たすべき機能や性能を明確にし、すべてのステークホルダーの合意を得て文書化するプロセス。
- `システム開発プロセス`: 合意された要件に基づき、システムを設計、構築、テストし、受け入れ可能な状態にするプロセス。

一方で、 `要求` と `要件` については明確に定義されておらず、いずれも英語における `Requirement` の訳語として、同等の言葉として扱われている。業界でも混用されることが多い。

本ガイドでは、IPA 共通フレーム 2013 のプロセス定義（企画プロセス / 要件定義プロセス）を参考に、下記の通り使い分けることする。

- `要求`: 事業側（発注者）が提示する `Requirement`。達成したい目的・課題、またはシステムへの期待。抽象度が高いインプット。
- `要件`: 技術側（受注者）が定義する `Requirement`。要求を分析し、システムが満たすべき具体的・検証可能な仕様として定義したもの。抽象度が低いアウトプット。

## [requirement: どのように要求を整理するか](./assets/01-requirement.md)

コンセプト: **_漠然とした要望や問題意識を収集し、課題をシンプルに構造化してコンセプトを打ち出す_**

トピック:

- 洞察と目標設定: 今はまだ誰もわかっていない「本当に必要なもの」をあぶりだす
- コンセプト策定: 解決策をコンセプトとストーリーで語り、ビジネス価値を訴求する
- スコープ定義: やらないことを定めることで、やることを浮き彫りにする

## [business: どのように業務要件を定義するか](./assets/02-business.md)

コンセプト: **_システムを取り巻く業務と登場人物をモデル化し、システムの役割を浮き彫りにする_**

トピック:

- ユビキタス言語定義: 十人十色に使われる一般用語を、十人一色にする
- アクターとペルソナのモデル化: 人も機械もモデル化し、シンプルな世界として考える
- ドメイン知識の構造化: 業務で扱うノウハウを整理・抽象化し、データ設計の基礎を築く
- 業務フローと例外設計: システムの外部の振る舞いを定めることで、システムに求められる振る舞いを浮き彫りにする

## [specification: どのようにシステム要件を定義するか](./assets/03-specification.md)

コンセプト: **_システムの振る舞いを定め、システムの取りうる姿を収束させる_**

トピック:

- 前提条件と制約条件の整理: プロジェクトが変えられないこと（コントロールできない外枠）を明確にする
- システム方式と境界設計: システムを構成する論理コンポーネントとそれらの責任範囲を定め、システムの骨格を形成する
- データ要件定義: 業務要件で定義したドメイン知識を、永続化可能な論理データモデルに落とし込む
- 機能要件定義: 業務上の振る舞いを、システム内部の具体的な処理ロジックと状態変化として定義する
- インターフェース要件定義: システムと外部（ユーザー、他システム）との間の「契約」を定める
- 非機能要件定義: 技術者でないとイメージしづらい機能以外の要求を、計測可能な品質特性として定義する
- 要件ドキュメント管理の心得: 要件と設計は不可分であることを前提に、要件定義書の限界との付き合い方を見極める
