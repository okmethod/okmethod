# organization: どのように組織構造の境界を設計するか

**コンセプト**: 適切なアーキテクチャスタイルと技術スタックの選定ポイントを押さえ、システムの物理的な境界線を慎重に決定する

---

## 関連キーワード

- **Monolith First** (モノリス・ファースト - 最初は一枚岩で作る)
- **Modular Monolith** (モジュラーモノリス - 内部で分割された一枚岩)
- **Microservices** (マイクロサービス - 独立してデプロイ可能な小さなサービス群)
- **Conway's Law** (コンウェイの法則 - システム構造は組織構造に従う)
- **Inverse Conway Maneuver** (逆コンウェイ戦略 - 望ましいアーキテクチャに合わせて組織を設計する)
- **Bounded Context** (境界づけられたコンテキスト - 論理的なモジュール境界・物理分割の候補)
- **Data Ownership** (データオーナーシップ - 各サービスが自身のデータを所有する)
- **On-Premises** (オンプレミス - 自社データセンターでのインフラ設計・運用)
- **Cloud-Native** (クラウドネイティブ - クラウドの機能を最大限に活かしたインフラ設計・運用)
- **Serverless** (サーバーレス - インフラ管理を委譲する)
- **Boring Technology** (退屈な技術 - 枯れた安定技術を選ぶ)
- **Innovation Tokens** (イノベーショントークン - 新しい技術への挑戦は数回に限る)
- **Event-Driven Architecture** (イベント駆動アーキテクチャ - 非同期メッセージングによる疎結合)
- **IaC** (Infrastructure as Code - インフラをコードで管理する)

## アーキテクチャスタイル選定の心得

**_組織の規模とフェーズに合わせて、システムの分割単位（デプロイメント単位）を決める。_**

> 主要なアーキテクチャスタイルの特性・トレードオフ一覧 → [アーキテクチャスタイル チートシート](./architecture-styles-cheatsheet.md)

### 複雑性の主戦場がどこにあるかを問う

- Organization レイヤーの設計エネルギーをどこに集中させるかは、複雑性の主戦場によって変わる
  - **チーム間コミュニケーションが主戦場**: サービス境界の引き方（コンウェイの法則への対処）が中心課題
  - **スケーラビリティが主戦場**: デプロイ単位の独立性と水平スケール設計が中心課題
  - **外部依存の変動が主戦場**: ベンダーロックイン対策とインフラの抽象化が中心課題
- 「とりあえずマイクロサービス」や「とりあえずクラウドネイティブ」はこの問いを飛ばした結果であり、過剰な複雑性を生む

### 安易にマイクロサービスを選ばない（Monolith First）

- スタートアップや新規事業の初期段階では、**「モジュラーモノリス」**を第一選択肢とする
  - 理由: ドメインの境界線が曖昧な状態でサービスを物理分割すると、分散モノリス（密結合な分散システム）になり、開発効率が劇的に低下するため
- マイクロサービスへの移行は、「組織が拡大し、コミュニケーションコストが限界に達した時」や「独立したスケーラビリティが必須になった時」に検討する

### 組織とアーキテクチャは鏡写しになる（Conway's Law）

- コンウェイの法則: 「システムの構造は、それを設計した組織のコミュニケーション構造に従う」
  - 3つのチームがあれば、3つのサブシステムが生まれる。これは意図した設計ではなく、組織の力学が自然に生み出す結果である
- 逆コンウェイ戦略: 「望ましいアーキテクチャを先に設計し、そのアーキテクチャが自然に生まれる組織構造を作る」
  - 「チームが分かれているからサービスを分ける」という順序を逆にすることで、組織の慣性にアーキテクチャが引きずられるのを防ぐ

### 論理境界（Bounded Context）が物理境界の候補になる

- DDD の Bounded Context（論理的なモジュール境界）が明確に引けたとき、そこがマイクロサービス分割の候補になる
  - 「注文ドメイン」「在庫ドメイン」が独立した Bounded Context として成立しているなら、それぞれを独立したサービスに切り出せる
  - Bounded Context の境界が曖昧なまま物理分割すると、サービス間の依存が密になり分散モノリスに陥る
- Bounded Context ≠ マイクロサービス（1対1にする必要はない）
  - 初期はモジュラーモノリスとして複数の Bounded Context を 1 つのデプロイ単位に収め、必要になったときに切り出す

### データオーナーシップを明確にする

- マイクロサービスでは「各サービスが自身のデータを所有し、他のサービスはそのデータに直接アクセスしない」が鉄則
  - 他のサービスのDBテーブルを直接 JOIN したり、データベースを共有したりしない
  - 他サービスのデータが必要なときは、そのサービスの公開 API 経由で取得する
- この原則を破ると、サービス間の結合がDBレベルで生まれ、独立したデプロイ・変更が不可能になる（分散モノリスの典型的な症状）
- モジュラーモノリスの段階でもこの原則を守っておくと、将来の切り出しコストが大幅に下がる

### オンプレミスかクラウドネイティブかを決める

- クラウドネイティブを第一候補とし、以下の制約がある場合のみオンプレミスを検討する
  - **データ主権・コンプライアンス**: 政府・金融・医療等でデータを国外・社外に出せない規制がある
  - **ネットワーク分離要件**: インターネット接続自体が許可されない環境（閉域網）
  - **大規模固定負荷**: 負荷変動が少なく長期安定稼働が前提の場合、オンプレミスの方が総コストが低くなる
- クラウドを選ぶ場合でも、ベンダーロックインのリスクを意識しておく
  - マルチクラウド戦略はコストと複雑性が上がるため、まず単一クラウドで始める

### インフラの抽象度を決める

- マネージドサービスを積極的に活用する
  - 「車輪の再発明」を避け、差別化につながらないインフラ管理はクラウドベンダーに委譲する
- 抽象度の高い選択肢（FaaS > PaaS > IaaS）から順に検討し、制約があるときだけ一段下げる
  - 検討順の例: サーバレス（Cloud Functions, Lambda 等） → コンテナ実行環境（Cloud Run, Fargate 等） → VM（Compute Engine, EC2 等）
  - 制約の例: 実行時間制限・コールドスタート許容不可・常駐プロセス・特殊なランタイム要件
- ただし、ベンダーロックインのリスクと移植性のトレードオフを理解しておく
  - コンテナ（Docker/Kubernetes）を採用する場合、運用複雑度が高く専任インフラエンジニアが必要になる可能性を考慮する

## 技術選定の心得

**_「新しさ」ではなく「信頼性」と「習熟度」で技術を選ぶ。_**

### まず制約条件を特定する

- 技術選定の前に、プロジェクトの外枠（予算・期間・技術的制約・法的制約）を明確にする
  - 詳細 → [前提条件と制約条件の整理の心得](../../planning-principles/assets/03-specification.md)
- 特にライセンス確認は最初に行う
  - OSS のライセンス（GPL、AGPL、MIT、Apache 等）は商用利用・改変・再配布の可否に直接影響し、比較検討後に発覚すると選定やり直しになる
  - 技術的な特性によって候補を比較検討したとしても、ライセンス都合で事実上一択に絞られることも多い

### 「退屈な技術」を選ぶ（Choose Boring Technology）

- 枯れた技術（Boring Technology）は、未知のバグが少なく、ドキュメントや知見が豊富であり、採用もしやすい
  - 流行の最先端技術（Hype）は、問題にぶつかった時の解決コストが高い
- 「イノベーショントークン」の概念を持ち、リスクを取りすぎないようにする
  - プロジェクトで導入する「新しい技術」は 2〜3 個まで（トークンの数）に制限する
  - ビジネスのコアとなる部分にのみトークンを使い、それ以外（DB、Web フレームワーク等）は標準的な技術を選ぶ

### コミュニティとエコシステムを確認する

- 以下の観点でライブラリやフレームワークを評価する
  - メンテナンス頻度（最終コミット日）
  - コミュニティの活発さ（GitHub Star 数、Issue の消化率）
  - 長期サポート（LTS）ポリシーの有無
  - 破壊的変更のポリシーの有無

## データストア選定の心得

**_業務ドメインのデータ特性を踏まえ、その手段として最適なデータストアを選ぶ。_**

### データモデルをある程度固めてから選定に入る

- データストアの選定は、データモデルの検討が先行していなければ成立しない
  - 「どのDBを使うか」を先に決め、データモデルをそれに合わせようとすると、業務要件にとって不自然な設計になる
  - 例: NoSQL ありきで設計すると、本来正規化すべきデータを非正規化し、整合性の管理が困難になる
- 要件定義フェーズで検討した論理データモデル・データアクセス特性を起点にする
  - 詳細 → [データ要件定義の心得](../../planning-principles/assets/03-specification.md)
- データモデルの検討が深まるほど、選定の根拠が明確になる
  - **整合性要件**: 強整合性が必要か、結果整合性で許容できるか（→ 03-mechanics.md 参照）
  - **アクセスパターン**: トランザクション処理（OLTP）か、分析・集計処理（OLAP）か
  - **データ構造**: 関係が複雑で正規化が必要か、スキーマが柔軟で変化しやすいか
  - **スケール要件**: 垂直スケールで十分か、水平スケールアウトが必要か

### データモデルの複雑さに合わせてデータストアを選ぶ

- データモデルが答えを示す — 「何を使いたいか」ではなく「データ特性」から選ぶ
  - 例: 関係が複雑・整合性が重要・ビジネスロジックが多い → **RDBMS**（PostgreSQL, MySQL 等）
  - 例: シンプルなドキュメント構造・CRUD 中心・スキーマが柔軟に変化する → **ドキュメント DB**（MongoDB 等）
- 「NoSQL の方がスケールする」という理由だけで選ぶのは早計
  - 多くのシステムは RDBMS + キャッシュ（Redis 等）の組み合わせでスケール問題を解決できる
  - NoSQL を選ぶ = 強整合性を諦めることが多く、結果整合性の許容が前提になる
- データ特性から判断がつかない場合、RDBMSは低リスクな第一候補
  - 強力な基盤を持つ「枯れた技術」であり、知見・ドキュメントが豊富

## サービス間通信プロトコル選定の心得

**_同期か非同期かを意識的に選び、サービス間の結合度をコントロールする。_**

### 同期通信（HTTP / gRPC）の使いどころ

- 呼び出し元が「即時の応答」を必要とする場合、または強整合性が求められる処理に選ぶ
  - 例: ユーザーがボタンを押した結果を画面に返す、リアルタイムなバリデーション
- シンプルで実装・デバッグしやすい反面、依存先がダウンすると呼び出し元も影響を受ける（強結合）
  - Circuit Breaker（→ 03-mechanics.md 参照）を組み合わせて障害の連鎖を防ぐ

### 非同期通信（メッセージキュー / イベント駆動）の使いどころ

- 呼び出し元が即時応答を必要とせず、結果整合性で許容できる処理に選ぶ
  - 例: 注文確定後のメール送信、在庫更新、ログの集約
- サービス間が疎結合になり、依存先がダウンしても呼び出し元は影響を受けない
- ただし、非同期化すると「いつ処理が完了するかわからない」という複雑性が生まれる
  - イベントのスキーマ管理（破壊的変更の扱い）が新たな設計課題になる
  - 分散トレーシングがないと、どこで詰まっているか追えなくなる（→ 03-mechanics.md 参照）

## インフラ構築の心得

**_手作業を排除し、再現可能なインフラを作ることをデプロイ戦略に組み込む。_**

### インフラをコード化する（IaC）

- 画面ポチポチ（GUI 操作）での環境構築を禁止し、Terraform や AWS CDK 等ですべてコード化する
  - 「誰がいつ何を変更したか」を追跡可能にし、環境（開発・検証・本番）間の差異をなくす
- インフラもアプリケーションコードと同様にレビューし、バージョン管理する

### ペットではなく家畜として扱う

- 特定のサーバーに愛着（名前）を持たず、いつでも廃棄・再作成できる状態（家畜）にする
  - サーバー内での手動設定や状態保持（Stateful）を避け、イミュータブル（不変）なインフラを目指す
