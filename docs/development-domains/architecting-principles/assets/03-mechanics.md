# mechanics: どのように品質特性のバランスをとるか

**コンセプト**: トレードオフの力学を理解し、システムに求められる品質特性(-ility)を実現する

---

## 関連キーワード

- **CAP Theorem** (CAP 定理 - 分散システムの限界)
- **Eventual Consistency** (結果整合性 - いずれ整合する)
- **Idempotency** (冪等性 - 何度実行しても結果は同じ)
- **Stateless** (ステートレス - サーバーが状態を持たない設計)
- **Cache-Aside** (キャッシュアサイド - アプリ側でキャッシュを制御する最も一般的な戦略)
- **Circuit Breaker** (サーキットブレーカー - 連続失敗で依存先への呼び出しを遮断)
- **Timeout** (タイムアウト - 応答待ちに上限を設け、リソース枯渇を防ぐ)
- **Retry** (リトライ - 一時的な失敗を再試行する。指数バックオフと上限回数が必須)
- **Bulkhead** (バルクヘッド - リソースプールを分離し、障害の波及を限定する)
- **Fallback** (フォールバック - 失敗時にキャッシュ・デフォルト値・縮退機能で代替応答する)
- **Observability** (オブザーバビリティ - 可観測性)
- **Distributed Tracing** (分散トレーシング - リクエストの追跡)
- **Health Check** (ヘルスチェック - 生存確認)
- **Zero Trust** (ゼロトラスト - 内部ネットワークも信頼しない)
- **Defense in Depth** (多層防御 - 複数の防御層を持つ)
- **Attack Surface** (攻撃面 - 外部に露出している箇所の総体)
- **Rate Limiting** (レート制限 - 過剰なリクエストを遮断)
- **Graceful Degradation** (グレースフルデグレデーション - 機能縮退しても継続稼働)
- **Graceful Shutdown** (グレースフルシャットダウン - 処理完了を待って安全に停止)
- **Chaos Engineering** (カオスエンジニアリング - 本番相当環境に意図的に障害を注入して回復性を検証する)

## 品質特性のトレードオフの心得

**_品質特性間のトレードオフを理解した上で、優先すべき品質特性を見極め、ツールより先に構造でその要件を満たす。_**

### 品質特性は互いにトレードオフの関係にある

- 品質特性（-ility）は独立して存在するのではなく、一方を高めると他方が犠牲になる競合関係にある
  - 可用性 vs 整合性: 冗長構成で可用性を高めると、ノード間のデータ同期タイミングで整合性が損なわれる
  - パフォーマンス vs 整合性: キャッシュで応答速度を上げると、データが古くなるリスクが生まれる
  - セキュリティ vs 利便性: 認証・認可のレイヤーを増やすと、ユーザー体験が重くなる
  - 変更容易性 vs パフォーマンス: 抽象化層を増やすほど変更は容易になるが、実行時オーバーヘッドが増える
- トレードオフを無視して「すべての品質特性を最高レベルで満たす」設計は、過剰に複雑なアーキテクチャになる
  - 品質特性の目標値はユースケースから引き出す — 詳細 → [非機能要件定義の心得](../../planning-principles/assets/03-specification.md)

### 分散システムでは CAP 定理がトレードオフの基本軸になる

- 分散システムでは「整合性（Consistency）」「可用性（Availability）」「分断耐性（Partition Tolerance）」の3つを同時に満たすことはできない（CAP 定理）
  - ネットワーク分断（Partition）は現実には避けられないため、実質的には「整合性」か「可用性」のどちらを優先するかの二択になる
  - **CP 系**（整合性優先）: 分断発生時に応答を止め、データの一貫性を保つ — 例: HBase, Zookeeper
  - **AP 系**（可用性優先）: 分断発生時も応答を返し、後からデータを整合させる（結果整合性）— 例: Cassandra, DynamoDB
- 多くの RDBMS は単一ノードでは CA（整合性 + 可用性）を保証するが、マルチノード構成では P（分断耐性）を選ばざるを得なくなる

### 「主役」から設計を始める

- 品質特性の設計は、「一覧を並べて優先順位をつける」のではなく、「最も制約が厳しく選択の余地がない箇所」を先に見つけることから始まる
  - 例: データ保全要件が厳しい → データモデルとデータストアが主役
  - 例: 重い処理が必要 → GPU 等のコンピューティングリソースが主役
  - 例: レスポンス速度が生命線 → コンピューティングとネットワークのレイテンシが主役
- 主役が決まると、それに合わせるように周辺の設計が自ずと決まる
  - 逆に「すべての品質特性を同時に最高レベルで満たす」設計を目指すと、過剰に複雑なアーキテクチャになる

### 品質特性はツールより先に構造で防ぐ

- 品質特性（整合性・可観測性・セキュリティ・回復性）は、ツールやパターンを「後から付け足す」ことで達成するものではなく、アーキテクチャの構造そのものから生まれる
  - 整合性: 分散トランザクションを必要とする構造を避けることで、「整合性を壊せない」状態を作る
  - 可観測性: 層構造が明確であれば、問題が起きたときの被疑箇所を事象から特定できる
  - セキュリティ: 露出ポイントを減らす構造にすることで、攻撃面を作らない
  - 回復性: 障害の連鎖を許す構造を避けることで、影響範囲を構造的に限定する
- ただし「良い構造があれば品質特性が自然に生まれる」は循環するため、主役（最大の制約）を先に決めることで構造の方向性を定める

## パフォーマンスとスケーラビリティの心得

**_計測なしに最適化せず、スケールの壁を構造で突破する。_**

### ボトルネックは仮説ではなく計測で特定する

- 「ここが遅い」という直感より、計測結果を優先する
  - 最適化の対象が間違っていると、コストだけかかって効果がでない
  - プロファイラ・クエリログ・分散トレースを使い、実際に時間がかかっている箇所を可視化する
- パフォーマンス目標は数値で定義する
  - 「速くする」ではなく「95パーセンタイルで 300ms 以内」のように、測定可能な目標を設定する
  - 詳細 → [非機能要件定義の心得](../../planning-principles/assets/03-specification.md)
- 計測でボトルネックを特定したら、実装レベルのアンチパターンを潰すことが次の手になる
  - 具体的なアンチパターン（N+1 問題、不必要な全カラム取得、同期処理の直列実行等）→ [コーディングの心得](../../coding-principles/README.md)

### キャッシュで読み負荷を逃がす

- 読み取りが多い箇所には、キャッシュ（Redis 等）を導入して DB への負荷を分散する
  - **Cache-Aside**: アプリがキャッシュを確認し、なければ DB から取得してキャッシュに格納する（最も一般的）
  - **Write-Through**: 書き込み時にキャッシュと DB を同時に更新する（整合性は高いが書き込みコストが増える）
  - これらは切り替え可能な Strategy として設計しておくと、後から戦略を変更しやすい
- TTL（有効期限）を適切に設定する
  - TTL が長すぎると古いデータが返り続け、短すぎるとキャッシュの効果がなくなる
- キャッシュは「パフォーマンス vs 整合性」のトレードオフそのもの — キャッシュを入れると「データが古くなるリスク」を受け入れることになる

### スケールアウトを前提に設計する

- **ステートレス設計**: アプリケーションサーバーにセッションや状態を持たせない
  - セッション情報は外部ストア（Redis 等）に保存し、どのサーバーにリクエストが来ても同じ応答を返せる状態にする
  - これにより、サーバーを増減させても動作が変わらないスケールアウト構成が実現できる
- **スケールアップとスケールアウトを使い分ける**
  - スケールアップ（垂直）: 1台のサーバーを強化する。上限があり、コストも高くなる
  - スケールアウト（水平）: サーバーを並列に増やす。ステートレス設計が前提
  - 初期はスケールアップで対応し、限界が見えたらスケールアウトに移行する順番が現実的

## 整合性と信頼性の心得

**_分散システムにおける「完全な整合性」を諦め、現実的な解を選ぶ。_**

### 整合性レベルはユースケースから引き出す

- ビジネス側に「強整合性が必要か」を直接聞くと必ず「必要」と返ってくる - ユースケースを掘り下げることで、結果整合性で許容できるケースを見極める
  - 詳細 → [非機能要件定義の心得](../../planning-principles/assets/03-specification.md)
- 整合性レベルが明確になってはじめて、分散トランザクションを避けるかどうかの判断ができる

### 分散トランザクションを避ける

- マイクロサービスや複数の DB を跨ぐトランザクション（2PC, XA）は、複雑でパフォーマンスが悪いため、極力避ける
- 原則として、1 つのトランザクションは 1 つのサービスの 1 つの DB 内で完結させる
  - どうしても整合性が必要な場合は、「結果整合性（Eventual Consistency）」を採用し、メッセージキューや Saga パターンを用いて非同期に整合させる

### 冪等性（Idempotency）を徹底する

- ネットワークは信頼できないため、リクエストのリトライは必ず発生する
- すべての更新系 API（POST/PUT）やメッセージ処理は、同じリクエストが 2 回来ても問題ないように実装する
  - 例: 注文 ID をキーにして、既に処理済みならスキップする

## 可観測性の心得

**_「何が起きているか」ではなく「なぜ起きているか」を知るための仕組みを入れる。_**

### アーキテクチャの明確さが可観測性の基盤になる

- ログやトレースツールを整備する前に、「問題が起きたときの被疑箇所を絞れる構造になっているか」を確認する
  - 層構造が明確であれば、事象からおよその被疑箇所は判断できる - ログはその仮説を裏付けるために使う
  - 境界のない構造（Big Ball of Mud）では、ログを見ても原因特定が困難になる
- ログに出してはいけない情報（個人情報、認証情報）を出さないことが先決
  - エラーハンドラと伝播の仕組みを整備し、「何が出力されるか」を出力設計の段階で制御する
  - ツールで後から隠すのではなく、構造として管理する

### ログだけでは不十分と知る

- ログ（個別イベント）、メトリクス（集計値）、トレース（リクエストの連鎖）の 3 本柱を、システム内イベントの Observer として独立して整備する
- 特に分散システム（マイクロサービス）では、1 つのリクエストが複数のサービスを通過するため、「分散トレーシング（Trace ID の伝播）」が必須となる
  - これがないと、どのサービスがボトルネックになっているか特定できない
- すべてのログに相関 ID（Correlation ID）を付与し、一連の処理の流れを串刺しで検索できるようにする
  - 実装の詳細 → [ロギングの心得](../../coding-principles/assets/03-system.md)

## セキュリティの心得

**_「安全である」と過信せず、「侵害される前提」で設計する。_**

### 露出ポイントを減らすことが最初の防御

- 「どう守るか」より先に「何を露出させるか」を絞る
  - 外部に公開するエンドポイント、内部間の通信経路、アクセス可能なポート——これらを最小限にすることで、攻撃面（Attack Surface）を構造的に削る
  - 防御できるのは露出しているものだけ - 露出を減らすことが防御コストを下げる
- 認証・認可の設計方針（JWT vs セッション・認証プロバイダ選定・マイクロサービスでの配置）は技術選定の文脈で扱う → [認証基盤選定の心得](./01-organization.md)

### ゼロトラストを前提にする

- 「社内ネットワーク（VPC 内）だから安全」という境界型防御の考えを捨てる
- サービス間の通信であっても、相互認証（mTLS 等）や都度の認可チェックを行い、最小権限の原則を徹底する

### 多層で守る（Defense in Depth）

- 境界だけを守る設計を避け、複数の独立した防御層を持つ
  - 外部から来るすべての入力（ユーザー入力・外部 API レスポンス）はバリデーション・サニタイズする
  - 認証・認可・レート制限・入力検証を Chain of Responsibility として積み重ね、各層が独立して機能することで、1 層が突破されても被害を最小化できる
- 過剰なリクエストや不正アクセスの被害を抑えるため、エンドポイントにレート制限（Rate Limiting）を設ける

## 回復性の心得

**_障害を前提とし、部分的な失敗がシステム全体に波及しないようにする。_**

### 復旧目標値から回復性設計を始める（RTO / RPO）

- RTO / RPO はビジネス側と合意する非機能要件 — 定義と目標値の決め方 → [非機能要件定義の心得](../../planning-principles/assets/03-specification.md)
- この目標値によってアーキテクチャの選択が変わる
  - RTO が短い → ホットスタンバイ（常時起動した待機系）が必要
  - RTO が長くてよい → コールドスタンバイや手動リストア手順で対応できる
  - RPO がゼロに近い → リアルタイムレプリケーション（同期レプリカ）が必要
  - RPO に余裕がある → 定期スナップショット（バックアップ）で対応できる

### 想定外の障害を前提に設計する

- 想定できる障害はすべてハンドリングする一方で、「想定外のことは必ず起こる」を前提として受け入れる
  - 完全な障害ゼロを目指すのではなく、障害が起きたときの影響を最小化・局所化する設計を優先する
  - 「何が起きても大丈夫な構造」より「何が起きても致命的にならない構造」を目指す
- エラーハンドリングの網羅性はテストで担保する
  - 正常系だけでなく、外部依存の失敗・タイムアウト・異常データなど、障害シナリオを明示的にテストケースに含める

### 文脈に合わせて最低限のパターンを選ぶ

- 外部 API・別サービスへの同期呼び出しがある場合、最低限 **Timeout + Circuit Breaker** を組み込む
  - Timeout がなければ、依存先の遅延がスレッドを占有し続け、自システムもハングする
  - Circuit Breaker がなければ、依存先がダウンしても呼び出し続け、カスケード障害に発展する
- Retry を追加するなら、冪等性（Idempotency）が前提条件として成立していることを確認する
  - 冪等性のない操作を Retry すると、二重課金・二重登録等の二重処理を引き起こす
- 複数の外部依存を持つなら、Bulkhead でリソースプールを分離する
  - 1 つの依存先の詰まりが他の処理を巻き込まないようにする
- Fallback はユーザーが直接触れる機能（フロント向け API）に絞って検討する
  - バックグラウンド処理（ログ集約・分析等）への Fallback は過剰になりやすい

### 回復性パターンを組み合わせる

- 障害を前提とした設計では、以下のパターンを多層的に組み合わせて対処する
  - **Timeout（タイムアウト）**: 依存先からの応答を無限に待たない。スレッドやコネクションの枯渇を防ぐ最初の砦
  - **Retry（リトライ）**: 一時的な失敗（ネットワーク瞬断等）は再試行する。ただし無制限なリトライは障害を悪化させるため、指数バックオフ（間隔を指数的に延ばす）と最大試行回数を必ず設定する。冪等性（→ 整合性と信頼性の心得）が保証されていることが前提
  - **Circuit Breaker（サーキットブレーカー）**: 連続失敗が閾値を超えたら回路を開き、依存先への呼び出しを即座に遮断してカスケード障害を止める（Closed / Open / Half-Open の State パターンによる状態遷移）
  - **Bulkhead（バルクヘッド）**: 機能ごとにスレッドプールやリソースを分離し、一部の依存先の遅延・停止が他の処理に波及しないようにする
  - **Fallback（フォールバック）**: 失敗時に代替応答を返す。キャッシュされた前回の結果・デフォルト値・縮退した機能などの戦略がある
- 単独ではなく組み合わせて使うのが基本: **Timeout → Retry → Circuit Breaker → Fallback** の順に多層で対処する

### 失敗できるようにする（Graceful Degradation）

- 一部の機能が壊れても、システム全体のメイン機能だけは動かし続ける
  - 例: 「おすすめ商品」サービスの応答がなくても、トップページや商品購入機能は表示させる（フォールバック処理）

### 安全に停止する（Graceful Shutdown）

- システム停止時やデプロイ時は、処理中のリクエストを完了させてから停止する
  - 強制終了すると、処理途中のトランザクションがロールバックされず、データ不整合や未応答が発生する
  - SIGTERM を受け取ったら新規リクエストの受け付けを止め、既存リクエストの完了を待ってから終了する

### 障害を意図的に注入して設計を検証する（Chaos Engineering）

- 「障害を前提とする設計」が実際に機能するかどうかは、本番で障害が起きるまでわからない — これを解消するのがカオスエンジニアリング
  - 本番（またはそれに近い環境）で、ネットワーク遅延・サービスダウン・リソース枯渇などを意図的に注入し、Timeout・Circuit Breaker・Fallback が設計どおりに動くかを検証する
  - テスト環境での動作確認と、本番相当の条件での動作は別物として扱う
- 小さく始める
  - 最初からカオスツール（Chaos Monkey, LitmusChaos 等）を導入する必要はない
  - 「特定のサービスを手動で停止してみる」「ネットワークに遅延を入れてみる」という単純な実験から始めることで、設計の盲点が見つかることが多い
