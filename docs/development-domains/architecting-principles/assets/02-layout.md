# layout: どのように内部構造を区画整理するか

**コンセプト**: コンポーネント内部の関心を分離し、ドメインの複雑性をシンプルに表現する

---

## 関連キーワード

- **Layered Architecture** (レイヤードアーキテクチャ)
- **Dependency Rule** (依存性のルール - 依存は外から内へ)
- **Clean Architecture / Onion Architecture** (クリーンアーキテクチャ / オニオンアーキテクチャ)
- **Package by Feature** (機能単位のパッケージング)
- **Vertical Slice Architecture** (垂直スライスアーキテクチャ - 機能ごとに層を貫通させる)
- **Bounded Context** (境界づけられたコンテキスト - モジュールごとに独立したモデルと言語を持つ)
- **DTO** (Data Transfer Object - データ転送用オブジェクト)
- **CQRS** (Command Query Responsibility Segregation - 書き込みと読み取りのモデルを分離するパターン)
- **Event Sourcing** (イベントソーシング - 状態をスナップショットではなくイベントの積み重ねで表現するパターン)

## レイヤー設計の心得

**_責務を分離し、依存の方向を制御して、変更容易性とテスト容易性を高める。_**

### 責務をレイヤーに分けて整理する（Layered Architecture）

- UI・ビジネスロジック・データアクセスを層（レイヤー）として分離し、各層が下位層にのみ依存するアーキテクチャスタイル（Layered Architecture）
  - 最もシンプルな構成は Presentation → Business Logic → Data Access の 3 層
  - 層を分けることで「どの層が何の責務を持つか」が明確になり、変更の影響範囲を層内に限定しやすくなる
- レイヤーの分離境界は、テストの分離点でもある
  - 各層を独立して検証できる設計が、テスト容易性の基盤になる
  - 「どの層で何をテストするか」の戦略 → [テスト設計の心得](../../testing-principles/README.md)

### 依存の方向を一方向にする（Dependency Rule）

- ドメイン層を中心に同心円状にレイヤーを重ね、外側（UI・インフラ）から内側（ドメイン）への依存のみを許可するアーキテクチャスタイル（Clean Architecture / Onion Architecture）
  - 内側（ドメイン）は外側（フレームワークや DB）のことを一切知らない状態を保つ
  - これにより、 DB を RDBMS から NoSQL に変えたり、Web フレームワークを差し替えたりしても、ビジネスロジックへの影響を最小限に抑えられる

### 各層の厚みは「複雑性の主戦場」によって決まる

- レイヤー設計の目的は、設計エネルギーを最も複雑な層に集中させること
  - ドメインルールが複雑なシステムでは、ドメイン層を厚くすることが正解（ルールエンジン化など）
  - 外部リソースが多様・変動しやすいシステムでは、インフラ層の Port/Adapter 境界を厳格に保つことが正解
- 「ドメイン層が薄い」「インフラ層がシンプル」は設計ミスではなく、複雑性の主戦場がそこではないことの反映

### 分離設計への投資は変更容易性として回収される

- 分離設計（Port/Adapter、Facade 経由の制約など）は実装時のコストだが、変更容易性として回収される
  - 変更容易性の本質は「どの層・どのモジュールを変えればいいかのスコープが明確で、変更が波及しない」こと
  - Port 境界を維持していれば、外部ライブラリを差し替えても影響範囲がインフラ層の数ファイルに限定される
- テスト容易性も分離設計に依存する。ロジックをドメイン層に寄せるほど、状態のないステートレスな単体テストが書きやすくなる

### 過剰な抽象化（Over-Engineering）を避ける

- 小規模なアプリケーションで厳格なクリーンアーキテクチャ（全レイヤーでの DTO 変換など）を採用すると、ボイラープレートコードが増えすぎて開発速度が落ちる
  - 単純な CRUD アプリであれば、MVC や 3 層アーキテクチャで十分な場合もある
- 過剰な抽象化はテストにも波及する — 不要な層やインターフェースが増えると、それを動かすためのモック・スタブも比例して増える（Over-Mocking）
- 「YAGNI 原則」に従い、必要になったタイミングでレイヤーを分割・抽象化する
  - テストが書けなくなったとき（外部依存を差し替えられない、副作用とロジックが混在しているなど）は、設計の限界を示すシグナル

## モジュール分割の心得

**_機能（ビジネスドメイン）単位でコードをまとめ、高凝集・低結合な構造を作る。_**

### 機能単位でパッケージングする（Package by Feature）

- `controllers/`, `models/`, `views/` のようなレイヤー単位の分割（Package by Layer）は、機能追加の際にあちこちのフォルダを開く必要があるため避ける
- `orders/`, `users/`, `products/` のように、関連する機能（ユースケース、ドメイン、UI）を 1 箇所にまとめる（Package by Feature / Modular Monolith）
  - これにより、ある機能を削除する際、そのフォルダを消すだけで済む（凝集度が高い状態）
- 「機能単位にまとめる」思想をさらに徹底したものとして、ユースケース単位でコードを完結させるスタイルもある（Vertical Slice Architecture）
  - レイヤー（水平方向）ではなくユースケース（垂直スライス）を単位として、1 つの機能に必要な処理をすべて 1 箇所に完結させる
  - あるユースケースの変更が他に波及しにくい反面、スライス間のコード重複は増える

### モジュール境界は「揺るがない本質」を軸に引く

- 境界の引き方には「ドメイン概念」「インフラ種別」など本質的な軸と、「ファイルサイズ」「実行タイミング」など技術的都合の軸がある
  - 本質的な軸で引いた境界はコードが増えても安定する
  - 技術的都合を軸にした境界は、機能追加のたびに揺らぎやすい
- 何を境界の軸にしているかを意識し、言語化できる状態を保つ

### 拡張の単位を言語化できることが、設計が機能している証拠

- 設計が機能しているかどうかは、「何を追加するときに、何をすればよいか」を一言で言えるかで確認できる
  - 言える例: 「ゲームを1本追加する = Facade + Store + UI のファイルセットを追加する」
  - 言えない例: 「機能を追加するたびに、全体を見直す必要がある」
- 拡張の単位が明確でない設計は、追加のたびに全体を把握し直すコストが生じ続ける

### 基底クラスより部分的なユーティリティ再利用を優先する

- 複数のモジュールで共通化できる部分が「一部のユーティリティにとどまる」場合、基底クラス（継承）より関数の再利用を選ぶ
  - 基底クラスは、継承できる範囲が狭いほど、ボイラープレートと継承ツリーの複雑さのコストだけが残る
  - ユーティリティ関数として切り出すだけで、モジュールごとの独立性が高まり、テストも書きやすくなる
- 「共通化できる量」と「継承コスト」を比べて、共通化できる量が少なければ継承しない

### コンテキストの境界を守る

- 各モジュールは、その内部でのみ通用するドメインモデルと言語を持つ（Bounded Context）
  - 「注文」という言葉も、販売コンテキストと在庫コンテキストでは意味が異なる場合がある
  - 境界を越えた概念の共有は、モデルの汚染や意図しない結合を生む
- 異なるモジュール（機能）間の依存は、定義された公開インターフェース経由のみに限定する
  - 他のモジュールのデータベーステーブルを直接 JOIN したり、内部クラスを直接 import したりしない
  - これを守ることで、将来的にそのモジュールをマイクロサービスとして切り出すことが容易になる
- Bounded Context を物理分割（デプロイ単位）に昇格させる判断基準 → [organization: 組織構造の境界設計](./01-organization.md)

## データモデル配置の心得

**_ドメインの知識をどこに置くかを明確にする。_**

### ドメインモデル貧血症を防ぐ

- エンティティ（データクラス）には、データだけでなく、そのデータを操作する振る舞い（メソッド）も実装する
  - getter/setter だけのクラスにせず、業務ルール（例: `can_cancel()`, `apply_discount()`）を持たせる
- サービス層（UseCase）が太りすぎないように、ロジックを可能な限りドメインモデルに寄せる

### データの変換（マッピング）戦略を決める

- レイヤーを跨ぐ際、データを詰め替えるか（マッピング）、そのまま通すか（共有）の方針を決める
  - 理想: 各層専用のモデル（DTO, Entity, DB Model）を用意し、変換する（結合度が低い）
  - 現実（妥協案）: シンプルなアプリなら、ドメインエンティティをそのまま UI や DB に渡すことを許容する

### 読み書きモデルを分離する（CQRS）

- 単一のデータモデルを読み取りと書き込みの両方に使うと、クエリ最適化とドメインモデルの整合性が競合することがある
  - 書き込み側: ドメインルール・整合性・トランザクションを重視した正規化モデル
  - 読み取り側: UI・レポートの表示効率を重視した非正規化ビュー（結合・集計済みのフラットなデータ）
- **CQRS（Command Query Responsibility Segregation）**: コマンド（書き込み）とクエリ（読み取り）のモデルを分離するパターン
  - 読み取り側を専用の非正規化ビューとして切り出すことで、クエリ最適化とドメインモデルの純粋性を両立できる
  - 複雑なドメインか、読み書きの性能要件が大きく異なる場合にのみ導入を検討する — 単純な CRUD には過剰
- **Event Sourcing**: 状態をスナップショットではなく「何が起きたか（イベント）」の積み重ねとして保存するパターン
  - 現在の状態はイベントの再生（Replay）から導出する
  - 完全な変更履歴が残り、過去の任意時点の状態を再現できる反面、実装複雑度が高く、イベントスキーマの後方互換性管理が課題になる
- CQRS と Event Sourcing は組み合わせて使われることが多いが、独立したパターンであり、片方だけ採用することも可能
