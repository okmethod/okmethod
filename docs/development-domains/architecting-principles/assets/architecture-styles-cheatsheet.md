# アーキテクチャスタイル チートシート

Organization 層の設計で参照する主要なアーキテクチャスタイルの索引。

**参考**: [O'Reilly「ソフトウェアアーキテクチャの基礎」](https://www.oreilly.co.jp//books/9784873119823/)

**ラベルの凡例**:

- `向くケース:` — このスタイルが力を発揮する場面
- `トレードオフ:` — 得るものと失うもの
- `落とし穴:` — よくある誤用の罠・限界に達したときの移行シグナル
- `注:` — 書籍・本ガイドにおける分類上の位置づけ

---

## モノリシック系（単一デプロイメント単位）

単一のデプロイ単位に機能をまとめるスタイル。シンプルさが強みだが、規模拡大に伴う課題がある。

### Monolithic

すべての機能を 1 つのデプロイ単位に収める

- 向くケース: 小規模・プロトタイプ・初期フェーズ
- トレードオフ: シンプルで開発が速いが、規模拡大するとデプロイ・変更が重くなる
- 落とし穴: 境界設計を怠るとモジュール境界のない無秩序な状態（「巨大な泥団子 / Big Ball of Mud」）になる

### Modular Monolith

内部をドメイン境界で明確に分割した単一デプロイ

- 向くケース: ドメイン境界は見えているが、物理分割するほどの規模・チームではない段階（スタートアップ〜中規模の成長期）
- トレードオフ: 将来のマイクロサービス化への移行コストを下げられるが、モジュール境界の維持に継続的な設計コストがかかる
- 落とし穴: モジュール間の依存を厳格に管理しないと内部で密結合が進み、通常のモノリスと変わらなくなる
- 注: 参考書籍には掲載されていないが、Layout 層寄り設計スタイル(後述)を単一デプロイ単位に適用したものとして広く使われる

### Microkernel

最小限のコアにプラグインで機能を追加する

- 向くケース: 製品ソフト・IDE・拡張性が重要なシステム
- トレードオフ: 拡張性が高いが、プラグイン間の整合性やコアの設計が難しい
- 落とし穴: コアへの依存が深まったプラグインが増えると差し替えが困難になる。またコアに機能を追加し続けると「プラグイン機構を持つだけのモノリス」になる

## モノリシック系（単一デプロイメント単位） - Layout 層寄り設計スタイル

アプリケーション内部の責務分割・依存方向を決めるスタイル。  
参考書籍ではアーキテクチャスタイルとして扱われているが、本ガイドの分類では [Layout 層](../README.md) の関心事に近い。

### Layered

UI・ビジネスロジック・データアクセス等の責務を水平な層に分割する

- 向くケース: 汎用的なエンタープライズシステム
- トレードオフ: 構造がシンプルで理解しやすいが、層をまたぐ変更が多いと結合度が上がりやすい
- 落とし穴: 横依存が増えてくると、ドメイン境界が曖昧になる。これは Clean / Hexagonal への移行を検討するシグナル

### Clean / Hexagonal (Port/Adapter)

同心円状構造で分離し、内側は外側の技術変化に影響されない

- 向くケース: 技術変化が激しい・ドメインロジックが複雑・外部依存を差し替えたい場面
- トレードオフ: ドメインが技術詳細から完全に独立するが、Port（インターフェース）の設計・レイヤー数が増え学習コストが上がる
- 落とし穴: Port/Adapter の層を増やすことが目的化し、ドメインが薄い CRUD アプリに過適用すると儀式的なボイラープレートが増える
- 注: 参考書籍には掲載されていないが、Layered の進化形として広く使われる

### Pipeline

データをフィルターのチェーンで順に変換するパイプ＆フィルター構造

- 向くケース: ETL・データ変換・バッチ処理
- トレードオフ: 処理ステップの追加・差し替えが容易だが、ステップ間の状態共有が難しい
- 落とし穴: ステップをまたぐエラーハンドリングや再処理を設計しないと、途中失敗時にどこから再開すべきか判断できなくなる

## 分散系（複数デプロイメント単位）

複数のデプロイ単位に分割するスタイル。独立性・スケーラビリティを得る代わりに、分散システムの複雑性を引き受ける。

### Service-Based

粗粒度のサービス（数個〜十数個）に分割する分散スタイル

- 向くケース: 中〜大規模の企業システム
- トレードオフ: マイクロサービスより運用しやすいが、DB を共有する構成が多く完全な独立デプロイが難しい
- 落とし穴: サービス間でDBを共有すると「分散モノリス」の亜種に陥る。また粒度をマイクロサービスに近づけるほどシンプルさのメリットが消える

### Event-Driven

イベントを非同期メッセージで伝播させ、コンポーネントを疎結合にする

- 向くケース: 高スループット・非同期処理・疎結合が重要な場面（大量イベントのSaaS・EC・配車、監査証跡が必要な金融など）
- トレードオフ: 疎結合・高スループット・障害分離を実現できるが、最終整合性・デバッグ・スキーマ管理が複雑になる
- 落とし穴: イベントスキーマへの後方非互換な変更が下流サービスに波及する。スキーマの破壊的変更ポリシーを決めていないと、サービス間に暗黙的な結合が生まれる。また強整合性が必須の処理（決済の確定など）に適用すると、整合性の保証が困難になる
- 注: CQRS・Event Sourcing と組み合わせることが多い（詳細 → [layout: データモデル配置](./02-layout.md)）

### Space-Based

インメモリデータグリッドをDB前段に置き、DBボトルネックを回避して高並行処理を実現する

- 向くケース: 極高負荷・高同時接続（チケット販売等）
- トレードオフ: 極めてスケールしやすいが、構造が複雑で採用難易度が高い
- 落とし穴: 実際の負荷を計測・検証せずに選択すると、導入・運用コストに見合わないオーバーエンジニアリングになる

### SOA

ESB（エンタープライズサービスバス）を介してサービスを連携するオーケストレーション型

- 向くケース: 大規模エンタープライズ・レガシー統合
- トレードオフ: 標準化・再利用性が高いが、ESB が単一障害点・ボトルネックになりやすい
- 落とし穴: ESB にビジネスロジックを実装し始める（Smart ESB）と、ESB への変更が全サービスに波及しボトルネックが加速する
- 注: SOA は本来より広い概念だが、参考書籍では ESB 中心の Orchestration-Driven SOA として整理されている

### Microservices

独立してデプロイ可能な小さなサービス群

- 向くケース: 大規模・複数チーム・独立スケールが必須な場面（ECプラットフォーム・決済基盤・動画配信など）
- トレードオフ: 高い独立性とスケール性を得るが、分散システムの複雑性（通信・観測・整合性）を全面的に引き受ける
- 落とし穴: 「テーブル単位（Userサービス・Orderサービス）」「技術単位（フロントエンドサービス・DBサービス）」での分割は Bounded Context を無視した典型的な誤りで「分散モノリス」を生む。またサービス数が増えて同期呼び出しが連鎖すると、単一障害点の増殖・カスケード障害のリスクが高まる。これは Event-Driven への移行を検討するシグナル

## 補足: コンピューティング・ITシステムの歴史

コンピューティングの歴史は大きく3つの時代で捉えられる。  
本資料における「アーキテクチャスタイル」は、このうちの **インターネット時代のみ** をスコープとしている。

| 時代                       | 期間                 | 代表技術・基盤         | 特徴・設計スタイル                       |
| -------------------------- | -------------------- | ---------------------- | ---------------------------------------- |
| メインフレーム時代         | 〜1980s              | IBM , COBOL, CICS      | 集中処理、バッチ処理                     |
| クライアント・サーバー時代 | 1980s後半〜2000s前半 | Unix, Windows, Oracle  | GUIクライアント＋DBサーバー（2〜3層C/S） |
| **インターネット時代**     | **1990s後半〜**      | HTTP, Java/.NET, Cloud | Webサービス（本資料にて紹介）            |

## 参考: アーキテクチャスタイルの発展の歴史

アーキテクチャの歴史は「複雑化する世界を分割・制御可能性を高める方向」への連続した発展の歴史。  
各スタイルは、前スタイルの矛盾・限界をきっかけとして、それを解決するために生み出されてきた。  
（参考: [弁証法](../../../specialty-domains/dialectics-engineering.md)）

| #   | スタイル                 | 時代        | 進化の動機                                                |
| --- | ------------------------ | ----------- | --------------------------------------------------------- |
| 0   | スクリプト型モノリス     | 〜1990s前半 | HTML・ロジック・DBが混在。分離の概念がない黎明期のWeb開発 |
| 1   | レイヤードアーキテクチャ | 1990s後半〜 | 企業システム化 → 責務を層で分けて「どこに書くか」を明確化 |
| 2   | クリーン/ヘキサゴナル    | 2000s後半〜 | 技術変化の激化 → ドメインを外側の技術から隔離             |
| 3   | サービス指向（SOA）      | 2000s〜     | レガシー統合 → 標準化・再利用性を求めてESBで束ねる        |
| 4   | マイクロサービス         | 2010s〜     | Web規模成長・チーム分散 → サービスを独立デプロイ単位に    |
| 5   | イベント駆動             | 2010s後半〜 | 非同期・高スループット → 状態でなく「変化の瞬間」を流す   |
| 6   | クラウドネイティブ       | 2010s後半〜 | 小粒度化した実行単位 → 障害前提の自動化・自己修復基盤     |
| 7   | AI エージェント          | 2020s〜     | LLM自律化 → エージェント行動の統制レイヤーが必須に        |

※番号は概念的・論理的な進化の順序。実際の年代は重複・並行する場合がある。  
（特に #2 はアプリ内部設計、#3 はシステム間統合と課題の軸が異なり、並行して発展した）

### #0 スクリプト型モノリス

CGI・初期 PHP/ASP スタイル。  
HTML・ビジネスロジック・DBアクセスが1ファイルに混在する形態で、設計を怠った結果ではなく、まだ責務分離の概念がなかった時代の素直な実装。  
機能追加のたびに全体を読む必要があり、規模が増すにつれ手に負えなくなる。

**限界**: 変更のたびに全体が影響を受け、やがて誰も全体を把握できなくなる。

**次への示唆**: 責務を層で分けて「どこに何を書くか」を明確にする必要が生まれる → #1 へ

### #1 レイヤードアーキテクチャ

> どこに書くか迷わないために分離する

責務を水平な層（Presentation / Application / Domain / Infrastructure）で分離し、依存は上から下への一方向のみに制限する。  
企業システムの複雑化に伴い登場し、小〜中規模に強いため、現在でも設計の出発点として有力。

**限界**: 規模が拡大すると Application 層が肥大化し、横の依存が増えてドメイン境界が曖昧になる。

**次への示唆**:

- アプリ内のビジネスロジックを技術的詳細から守りたくなったら → #2 へ
- 複数の既存システムを標準化・再利用したくなったら → #3 へ

### #2 クリーンアーキテクチャ / ヘキサゴナルアーキテクチャ

> 内側は外側を知らない。外側が内側に合わせる

Port（インターフェース）を境界に置き、DB・UI・フレームワークを差し替えてもドメインは無傷のまま保てる。  
Core（ドメイン＋ユースケース）は技術変化に影響されず、Adapter（UI / DB / 外部API）はいくつでも追加・差し替え可能。  
外部技術が急速に変わる環境ほど価値が高まる。ドメイン境界が整理できていれば、規模によってはモジュラーモノリスに留まることが最善のケースも多い。

**限界**: あくまで単一デプロイ単位であり機能単位での独立デプロイができないため、チーム分割やスケールが難しい。

**次への示唆**: チームや機能が拡大し、独立したデプロイ・スケールが必要になったら → #4 へ

### #3 サービス指向（SOA）アーキテクチャ

> 標準化・再利用性を求めてサービスを束ねる

ESB（エンタープライズサービスバス）を中心に、サービスを再利用可能な部品として標準化されたプロトコルで連携させる。  
エンタープライズのレガシー統合ニーズに応えたマイクロサービスの前身とも言われ、過渡期を担ったスタイル。ただし、レガシー統合用途では現在も現役。

**限界**: ESBへの集中が単一障害点・ボトルネックを生む。ESBにビジネスロジックが混入すると（Smart ESB）変更の波及が全サービスに及ぶ。

**次への示唆**: ESBに依存しない独立したサービス設計・デプロイが求められるようになったら → #4 へ

### #4 マイクロサービスアーキテクチャ

> 分散は目的ではなく結果。必要だから分ける

Bounded Context を単位に独立デプロイ可能なサービスに分割する。  
分割の動機は、「特定機能のみスケール」「チーム競合の回避」「変更影響の最小化」の3つが揃ったときに自然に生まれる。

**限界**: サービス数が増え同期呼び出しが連鎖すると、カスケード障害リスクが高まる。ネットワーク障害・整合性・監視の複雑性を全面的に引き受ける覚悟が必要。

**次への示唆**: 同期呼び出しの連鎖による障害伝播が問題になったら → #5 へ

### #5 イベント駆動アーキテクチャ

> 状態ではなく、状態が変化したことを通知する

「状態」でなく「変化した瞬間（イベント）」を非同期メッセージとして流し、サービス間を疎結合にする。  
EDA・CQRS・Event Sourcing の3つを組み合わせることで、疎結合・高スループット・監査証跡を同時に実現できる。

- EDA: イベントを中心に非同期でサービスを連動
- CQRS: 書き込みモデルと読み取りモデルを分離し、それぞれ独立にスケールさせる
- Event Sourcing: 現在の状態でなく、変化の履歴そのものを永続化する

**限界**: 最終整合性・デバッグ困難・イベントスキーマ管理の複雑さを引き受ける必要がある。強整合性が必須の処理（決済の確定など）には不向き。

**次への示唆**: 細粒度化した処理を効率的・自動的に動かす基盤が必要になったら → #6 へ

### #6 クラウドネイティブ

> 壊れることを前提とし、自動化・自己修復を設計する

「クラウド上で動かす」だけではなく、クラウドの伸縮・自動化・API主体の世界に合わせてアプリ自体を再構成することが本質。  
マイクロサービス・イベント駆動による実行単位の細粒度化を受け、それを効率的に動かす基盤として登場した。  
Serverless と Kubernetes を用途で使い分ける。

|              | Serverless        | Kubernetes         |
| ------------ | ----------------- | ------------------ |
| 最小単位     | 関数              | Pod                |
| 自動スケール | 手軽              | 設計が必要         |
| 運用負荷     | 非常に軽い        | チーム運用前提     |
| 向く用途     | Webhook・試作検証 | 常時稼働の本番環境 |

**限界**: Kubernetes はチーム運用が前提でスキル・運用コストが高い。インフラ知識だけでは不十分で、アプリ設計思想の転換が必要。

**次への示唆**: アプリの主体が AI エージェントになると、既存の認証・認可ではエージェント行動の統制ができなくなる → #7 へ

### #7 AI エージェント (Latest)

> エージェント行動を統制する

LLMの自律化によりエージェントが24時間稼働し、人間の数十〜数百倍規模のAPI呼び出しを行う時代。  
従来の API Gateway + IAM では「誰が・何を・どの許可で動かすか」と「暴走した場合の停止」を制御できず、制御プレーン（Identity / Authorization / Tool Governance / Kill Switch）を設計段階から組み込むことが必須になった。  
各社のアプローチは異なるが「統制レイヤーが必要」という方向性は一致している。

| 層                          | 役割                                     |
| --------------------------- | ---------------------------------------- |
| Identity                    | エージェントに固有IDを付与して身元を管理 |
| Authorization               | API単位でなくaction単位の権限制御        |
| Tool Governance             | ツール呼び出し引数の制約・検証           |
| Observability & Kill Switch | 行動の可視化と暴走時の停止機構           |

MCP がツール呼び出しの標準化候補として台頭しているが、日進月歩で変化が激しい。

**限界**: 業界標準が収束しておらず各社のアプローチも発展途上。既存の API Gateway + IAM との統合方法も確立されていない。

**次への示唆**: （次の矛盾がどこから顕在化するかはまだ見えていない）
