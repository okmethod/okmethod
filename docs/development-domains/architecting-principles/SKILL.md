# アーキテクチャ設計の心得

## 目的

システムのアーキテクチャ設計・レビュー・技術選定の意思決定をサポートする。  
「正解のアーキテクチャは存在しない」を前提に、**複雑性の主戦場を見極め、トレードオフを踏まえた特殊解を導き出す**ことを支援する。

## アセット

このスキルのナレッジファイルは `assets/` に配置されている:

- **`assets/01-organization.md`** — 物理境界・技術選定・インフラ戦略
- **`assets/02-layout.md`** — レイヤー・モジュール・データモデル
- **`assets/03-mechanics.md`** — 品質特性・整合性・可観測性
- **`assets/architecture-styles-cheatsheet.md`** — アーキテクチャスタイル比較表
- **`assets/GoF-design-patterns-cheatsheet.md`** — GoF デザインパターン一覧
- **`assets/clean-architecture-case-study.md`** — Clean Architecture 実践例

## 発動タイミング（TRIGGER）

以下のキーワード・文脈が含まれる場合に発動する:

- アーキテクチャ設計・見直し・レビューの相談
- 技術スタック・データストア・通信方式・認証基盤の選定
- モノリス / マイクロサービス / モジュラモノリスの比較
- 品質特性（可用性・整合性・パフォーマンス・セキュリティ等）のトレードオフ判断
- ADR（Architecture Decision Record）の作成
- 既存アーキテクチャの問題分析・ボトルネック特定
- レイヤー設計・モジュール分割・Clean Architecture の相談

## スキップ条件（SKIP）

以下の場合はこのスキルを使用しない:

- コーディング実装・リファクタリングの具体的な方法（→ コーディングの心得を参照）
- テスト設計・テスト戦略（→ テスト設計の心得を参照）
- 要件定義・ユーザーストーリー作成（→ 要件定義の心得を参照）
- 特定ライブラリ・フレームワークの API の使い方

## 自律実行ポリシー

- **コンテキストから推論できる場合は自律判断する** — ユーザーへの確認は最小限にとどめる
- **確認が必要な場合** — 回答の方向性が変わる制約条件が不明な場合のみ質問する
- **質問はまとめて 1 ターンで行う** — 必要な確認事項は一度にまとめ、往復を最小化する（最大 4 問）

## 実行手順

### Step 1: 問いを分析する

コンテキストから自律推論する。推論できない場合のみユーザーに確認する（1 ターンにつき最大 4 問）:

1. **対象スコープ**: システム全体 / 単一サービス / 単一コンポーネント（コード・文脈から推論）
2. **問いの階層**（キーワードから自律推論。判断が難しい場合は独自用語（Organization/Layout/Mechanics）を使わず確認する。例：「システム全体の構成についてですか？特定サービスの内部設計についてですか？」）:
   - **Organization** — 物理境界・技術選定・インフラ戦略。変更コスト: 特大。キーワード: モノリス/マイクロサービス、DB選定、認証基盤、クラウド
   - **Layout** — 単一コンポーネントの内部構造・依存関係。変更コスト: 大。キーワード: レイヤー設計、モジュール分割、Clean Architecture、DDD
   - **Mechanics** — 品質特性（-ility）・横断的関心事。変更コスト: 中〜特大。キーワード: パフォーマンス、整合性、可観測性、セキュリティ、回復性
3. **制約条件**: 既存スタック・チーム規模・スケール要件・期限（不明な場合のデフォルト前提: スタートアップ初期〜中期のWebサービス）

### Step 2: 対応アセットを読み込み、関連セクションを特定する

問いの階層に応じたファイルを **Read ツールで** 読み込む。複数の階層にまたがる場合は、**変更コストの大きい上位階層から順に**（Organization → Layout → Mechanics）読み込む:

| 問いの階層 | 読み込むファイル |
|---|---|
| Organization（物理境界・技術選定・インフラ） | `assets/01-organization.md` |
| Layout（レイヤー・モジュール・データモデル） | `assets/02-layout.md` |
| Mechanics（品質特性・整合性・可観測性等） | `assets/03-mechanics.md` |

チートシートは以下の条件で参照する:

- スタイル比較・選定の問いが出た場合 → `assets/architecture-styles-cheatsheet.md`
- GoF パターン名が会話に登場した場合 → `assets/GoF-design-patterns-cheatsheet.md`
- Clean Architecture / Hexagonal / Onion の実装例が求められた場合 → `assets/clean-architecture-case-study.md`

**読み込んだ後の処理**:

1. ユーザーの問いと照合し、関連するセクション・トピックを特定する
2. 関係ないセクションは読み飛ばし、関連箇所のみ抽出する
3. 抽出した内容を Step 3〜4 の回答根拠として使う

### Step 3: 関連トピックを提示する

**問いが 1 つのトピックに絞られている場合は Step 3 をスキップして直接 Step 4 へ進む。**

複数トピックにまたがる場合のみ、アセット内から関連するトピックを **2〜4 件** ピックアップして提示する。  
選定基準: **① 直接関連するもの優先、② 変更コストの大きい階層（Organization > Layout > Mechanics）を先に、③ 最大 4 件に絞る**。  
ユーザーが深掘りしたいトピックを選べるようにすることが目的。

**トピックマップ（変更コストの大きい順に並べている）**:

| キーワード例 | 参照トピック | ファイル |
|---|---|---|
| モノリス / マイクロサービス / スタイル比較 | アーキテクチャスタイル選定の心得 | `01-organization.md` |
| DB / データストア / RDB / NoSQL | データストア選定の心得 | `01-organization.md` |
| REST / gRPC / GraphQL / メッセージング | サービス間通信プロトコル選定の心得 | `01-organization.md` |
| 認証 / 認可 / SSO / JWT / OAuth | 認証基盤選定の心得 | `01-organization.md` |
| Kubernetes / クラウド / IaC / インフラ | インフラ構築の心得 | `01-organization.md` |
| レイヤー / 依存関係 / DIP | レイヤー設計の心得 | `02-layout.md` |
| モジュール / パッケージ / 分割 | モジュール分割の心得 | `02-layout.md` |
| ドメインモデル / エンティティ / DDD | データモデル配置の心得 | `02-layout.md` |

| セキュリティ / 脆弱性 / ゼロトラスト | セキュリティの心得 | `03-mechanics.md` |
| 可観測性 / ログ / メトリクス / トレース | 可観測性の心得 | `03-mechanics.md` |

| デザインパターン / GoF / Singleton / Strategy | GoF デザインパターン チートシート | `GoF-design-patterns-cheatsheet.md` |

### Step 4: トレードオフを列挙して推奨を出す

- 選択肢を 2〜3 個挙げる
- 各選択肢の「得るもの」と「失うもの」を明示する
- 「最善解」ではなく「**最悪回避**」の観点で推奨を出す

### Step 5: 意思決定を記録する

以下のいずれかに該当する場合に ADR を生成する:

- ユーザーが明示的に ADR 作成を依頼した場合
- Step 4 で Organization 階層の選択など、後戻りコストの大きい決定について、ユーザーが受諾・確定を示す発言をした場合

「なぜこの設計にしたか」と「何を諦めたか」をセットで残す。

```
### 推奨

**[推奨案]** を推奨する。理由: [根拠]

### 諦めること

[この選択で手放すもの]

### 関連トピック

詳しく掘り下げたい場合は以下を参照:
- [トピック名] → `ファイル名`
```

### 設計レビュー

```
## レビュー結果

### 問題点

- [どの原則が守られていないか・具体的な箇所と理由]

### 代替案

[より適切な設計と根拠]

### 関連トピック

- [トピック名] → `ファイル名`
```

### ADR（意思決定記録）

```
## ADR: [タイトル]

- **Status**: Accepted / Proposed / Deprecated
- **Context**: [背景・制約]
- **Decision**: [決定した内容]
- **Consequences**: 得るもの: [得るもの] / 失うもの: [失うもの]
```

## 設計判断の軸

1. **複雑性の主戦場はどこか** — 設計エネルギーを集中すべき場所を特定する
2. **変更コストの大きい決定から先に** — Organization > Layout > Mechanics の順で整理する
3. **最悪回避** — 最善解より「後悔しにくい選択」を優先する
4. **決定を記録する** — 将来の変更コストを下げるために、理由と諦めたものを残す
