# コーディングの心得

## 大原則：理解しやすいコードを書く！

- コードは「書かれる」よりも「読まれる」方が多い
- コードは他の人(未来の自分を含む) が最短時間で理解できるように書く
- コードを短く書くことは重要だが、理解にかかる時間を短くすることを優先する
- このコードは理解しやすいだろうか？と自問自答する

---

## 前提：本ガイドにおけるスコープとスタンスについて

本ガイドは、読みやすさ（`code`）・シンプルさ（`logic`）・実用性（`system`）の3層でコード品質を整理している。

本ガイドのスコープは、個々のコード片・関数・モジュールレベルの実装判断とし、モジュール間の依存関係・レイヤー設計・設計パターンの選択など、より上位レベルの判断は [アーキテクチャ設計の心得](../architecting-principles/README.md) および [テスト設計の心得](../testing-principles/README.md) が担う。  
また、言語横断的な原則として整理しており、言語固有のイディオムや慣習は対象外とする。

コーディングの領域は、リーダブルコードをはじめとする先人の知恵が強力に機能する「守」の分野である。
本ガイドはこれらの原則を整理し、「なぜそのルールがあるのかを理解した上で実践する」ことを目標とする。

## [code: どのようにコードを書くか](./assets/01-code.md)

コンセプト: **_個々のコード片を読みやすく書く_**

トピック:

- 命名: 命名により、その目的や内容が一目でわかる程度に明確にする
- コメント: コメントにより、コードからは読み取れない意図や仕様を伝える
- コードスタイル: コーディングスタイルを統一し、読みやすく管理しやすい状態にする

## [logic: どのように機能を書くか](./assets/02-logic.md)

コンセプト: **_複雑な問題を、シンプルに解けるように書く_**

トピック:

- 変数とスコープ: 変数を扱いやすくし、コードの見通しを良くする
- 式と演算: 複雑な式を分割し、一目で理解できるようにする
- 制御フロー: 制御フローを単純にし、読み手が立ち止まったり読み返したりしなくて済むようにする
- 機能の分割と配置: 関数やクラスは、簡潔かつ明確で見通しのよい設計にする
- アルゴリズム選択: 問題を効率的に解くアルゴリズムやデータ構造を選択する

## [system : どのようにシステムを書くか](./assets/03-system.md)

コンセプト: **_現実世界で動かしやすい実用的なシステムとして書く_**

トピック:

- エラーハンドリング: 想定外を想像し、あらかじめ備えることで想定内にする
- ロギング: いざとなった時に利用しやすいログを出力する
- パフォーマンス: 振る舞いは同じでも、スマートな方法を選ぶ
- セキュリティ: 攻撃者の視点で抜け道を考え、あらかじめ塞いでおく
- テスト: 作ってからテストを考えるのではなく、どのようにテストするか考えながら作る
