# コーディングの心得

## 目的

コード品質の向上・コードレビュー・リファクタリングの意思決定をサポートする。  
「理解しやすいコードを書く」を大原則に、**読みやすさ・シンプルさ・実用性の 3 層でコードを整理し、具体的な改善指針を導き出す**ことを支援する。

## アセット

このスキルのナレッジファイルは `assets/` に配置されている:

- **`assets/01-code.md`** — 命名・コメント・コードスタイル
- **`assets/02-logic.md`** — 変数・式・制御フロー・機能分割・アルゴリズム選択
- **`assets/03-system.md`** — エラーハンドリング・ロギング・パフォーマンス・セキュリティ・テスト
- **`assets/naming-cheatsheet.md`** — 命名パターンチートシート
- **`assets/cohesion-coupling-cheatsheet.md`** — 凝集度・結合度チートシート

---

## 発動タイミング（TRIGGER）

以下のキーワード・文脈が含まれる場合に発動する:

- コーディング実装・コードレビューの相談
- 命名・コメント・コードスタイルの改善
- 変数・スコープ・制御フロー・関数分割のリファクタリング
- エラーハンドリング・ロギング・パフォーマンス・セキュリティの実装方針
- 「このコードをどう書くか」「このコードは読みやすいか」の相談

## スキップ条件（SKIP）

以下の場合はこのスキルを使用しない:

- モジュール間の依存関係・レイヤー設計・設計パターンの選択（→ アーキテクチャ設計の心得を参照）
- テスト設計・テスト戦略（→ テスト設計の心得を参照）
- 要件定義・ユーザーストーリー作成（→ 要件定義の心得を参照）
- 特定ライブラリ・フレームワークの API の使い方

## 自律実行ポリシー

- **コンテキストから推論できる場合は自律判断する** — ユーザーへの確認は最小限にとどめる
- **確認が必要な場合** — 回答の方向性が変わる制約条件が不明な場合のみ質問する
- **質問はまとめて 1 ターンで行う** — 必要な確認事項は一度にまとめ、往復を最小化する（最大 4 問）
- **アセット外の回答は明示する** — 問いがアセットの記載範囲を超える場合、アセット内に根拠のある回答と、アセット外の一般論に基づく回答を明確に区別して説明する

## 実行手順

### Step 1: 問いを分析する

コンテキストから自律推論する。推論できない場合のみユーザーに確認する（1 ターンにつき最大 4 問）:

1. **対象スコープ**: 関数 / クラス / モジュール / ファイル単位（コード・文脈から推論）
2. **問いの層**（キーワードから自律推論）:
   - **code** — 読みやすさ。変更コスト: 小。キーワード: 命名、コメント、コードスタイル、フォーマット
   - **logic** — シンプルさ。変更コスト: 中。キーワード: 変数、スコープ、制御フロー、条件分岐、関数分割、アルゴリズム
   - **system** — 実用性。変更コスト: 中〜大。キーワード: エラーハンドリング、ログ、パフォーマンス、セキュリティ、テスタビリティ
3. **制約条件**: 使用言語・チームの慣習・パフォーマンス要件（不明な場合のデフォルト前提: 汎用的な Web アプリケーション）

### Step 2: 対応アセットを読み込み、関連セクションを特定する

問いの層に応じたファイルを **Read ツールで** 読み込む。複数の層にまたがる場合は、**変更コストの大きい上位層から順に**（system → logic → code）読み込む:

| 問いの層 | 読み込むファイル |
|---|---|
| system（エラー・ログ・パフォーマンス・セキュリティ） | `assets/03-system.md` |
| logic（変数・制御フロー・関数分割） | `assets/02-logic.md` |
| code（命名・コメント・スタイル） | `assets/01-code.md` |

チートシートは以下の条件で参照する:

- 命名パターン・変数名・関数名の命名相談が出た場合 → `assets/naming-cheatsheet.md`
- 凝集度・結合度・モジュール設計の話題が出た場合 → `assets/cohesion-coupling-cheatsheet.md`

**読み込んだ後の処理**:

1. ユーザーの問いと照合し、関連するセクション・トピックを特定する
2. 関係ないセクションは読み飛ばし、関連箇所のみ抽出する
3. 抽出した内容を Step 3〜4 の回答根拠として使う

### Step 3: 関連トピックを提示する

**問いが 1 つのトピックに絞られている場合は Step 3 をスキップして直接 Step 4 へ進む。**

複数トピックにまたがる場合のみ、アセット内から関連するトピックを **2〜4 件** ピックアップして提示する。  
選定基準: **① 直接関連するもの優先、② 変更コストの大きい層（system > logic > code）を先に、③ 最大 4 件に絞る**。  
ユーザーが深掘りしたいトピックを選べるようにすることが目的。

**トピックマップ（変更コストの大きい順に並べている）**:

| キーワード例 | 参照トピック | ファイル |
|---|---|---|
| エラーハンドリング / 例外 / エラー処理 | エラーハンドリングの心得 | `03-system.md` |
| ログ / ロギング / 出力 | ロギングの心得 | `03-system.md` |
| パフォーマンス / 速度 / 最適化 | パフォーマンスの心得 | `03-system.md` |
| セキュリティ / 脆弱性 / バリデーション | セキュリティの心得 | `03-system.md` |
| テスト / テスタビリティ / モック | テストの心得 | `03-system.md` |
| 関数 / クラス / 分割 / 責務 | 機能の分割と配置の心得 | `02-logic.md` |
| 制御フロー / 条件分岐 / ネスト / 早期リターン | 制御フローの心得 | `02-logic.md` |
| 変数 / スコープ / 副作用 | 変数とスコープの心得 | `02-logic.md` |
| アルゴリズム / データ構造 / 計算量 | アルゴリズム選択の心得 | `02-logic.md` |
| 凝集度 / 結合度 / モジュール | 凝集度・結合度チートシート | `cohesion-coupling-cheatsheet.md` |
| 命名 / 変数名 / 関数名 / クラス名 | 命名の心得 | `01-code.md` |
| コメント / ドキュメント / 説明 | コメントの心得 | `01-code.md` |
| フォーマット / インデント / スタイル | コードスタイルの心得 | `01-code.md` |
| 命名パターン / prefix / suffix | 命名パターンチートシート | `naming-cheatsheet.md` |

### Step 4: 具体的な改善指針を提示する

- 現状の問題点を指摘する（原則のどれが守られていないか）
- 改善案を示す（コードスニペットで示せる場合は示す）
- 「読む人が 0.1 秒でも早く理解できるか」の観点で評価する

## アウトプット形式

### コードレビュー

```
## レビュー結果

### 問題点

- [層: code / logic / system] [どの原則が守られていないか・具体的な箇所と理由]

### 改善案

[改善後のコードスニペットと根拠]

### 関連トピック

- [トピック名] → `ファイル名`
```

### 実装提案 / リファクタリング提案

```
## [実装提案 / リファクタリング提案]

### Before（リファクタリングの場合）

[現状のコード]

### After

[提案するコード]

### 理由

[なぜこの実装・変更で読みやすさ・シンプルさ・実用性が向上するか]
```

## コード品質の判断軸

1. **読む人が最短時間で理解できるか** — 「短く書く」より「理解コストを下げる」を優先する
2. **変更コストの大きい層から整理する** — system > logic > code の順で問題を整理する
3. **読みにくいコードを見たら設計レベルの問題を疑う** — コードは設計の結果であり、命名や構造の違和感は責務の設計ミスを示していることが多い
4. **原則は道具であり、目的ではない** — なぜそのルールがあるかを理解した上で適用する
